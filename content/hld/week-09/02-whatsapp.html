WhatsApp / Chat System Design

Designing a real-time messaging system at WhatsApp scale involves managing 500 million concurrent WebSocket connections, guaranteed message delivery, and presence tracking across thousands of servers.

Define: Requirements

  • Functional: 1:1 messaging, group chat (up to 256 members), delivery receipts (sent/delivered/read), online/offline presence, media sharing, end-to-end encryption
  • Non-Functional: Real-time (<1 second delivery), reliable (no message loss), ordered messages per conversation, 500M concurrent connections
  • Out of Scope: Voice/video calls, status/stories, payments

Estimate

# 2B registered users, 500M DAU
# Each user sends ~40 messages/day

# MESSAGES:
#   500M x 40 = 20B messages/day
#   20B / 86400 = 230K messages/sec (peak ~700K/sec)

# CONNECTIONS:
#   500M concurrent WebSocket connections
#   Each connection: ~10 KB memory overhead
#   500M x 10 KB = 5 TB of connection state
#   Need ~5,000 chat servers (1M connections each)

# STORAGE:
#   Per text message: ~200 bytes
#   Daily: 18B x 200B = 3.6 TB/day (Cassandra)
#   Media: 2B msgs x 2MB = 4 PB/day (S3, only URL in DB)

Architecture

# Message flow:
# 1. Client A sends message via WebSocket to Chat Server CS-1
# 2. CS-1 looks up recipient B's connected server (CS-3) via
#    Session Service (Redis: user_id -> server_id)
# 3. CS-1 routes message to CS-3 (server-to-server)
# 4. CS-3 pushes message to Client B via WebSocket
# 5. If B is offline -> message queued in Offline Store
#    When B connects, queued messages are delivered

# Delivery receipts:
#   SENT:      server received message (single check)
#   DELIVERED: recipient device received (double check)
#   READ:      recipient opened conversation (blue double check)

Key Components

  • Chat Servers: Each holds ~1M WebSocket connections. Stateful -- a user stays connected to one server.
  • Session Service (Redis): Maps user_id to their current chat server. Updated on connect/disconnect.
  • Message Queue: Kafka for reliable message routing between servers.
  • Message Store (Cassandra): Partitioned by conversation_id, sorted by timestamp. Optimized for sequential reads.
  • Offline Message Store: Queue per offline user. Drained on reconnect.
  • Presence Service: Heartbeat-based online/offline tracking with last-seen timestamps.

Group Messaging

  • Group message triggers fan-out to all members (up to 256)
  • Each member's chat server receives a copy
  • Delivery receipts are per-member in groups
  • Group metadata stored separately; message content stored once, referenced by all members

Design Considerations

  • End-to-end encryption: Server cannot read message content; only routes encrypted blobs
  • Message ordering: Sequence numbers per conversation ensure correct order even with network delays
  • Reconnection: Client sends last-received sequence number; server sends all messages after that
  • Presence optimization: Only notify close contacts of online/offline changes to reduce fan-out
