Notification System at Scale

A notification system delivers push notifications, emails, SMS, and in-app messages at billions per day. The challenge is reliability, user preferences, priority handling, and integration with multiple third-party delivery providers.

Define: Requirements

  • Functional: Multi-channel delivery (push via APNs/FCM, email via SES/SendGrid, SMS via Twilio, in-app via WebSocket), user preferences per channel per type, template system, priority levels (urgent/normal/low), deduplication, rate limiting
  • Non-Functional: Reliable (no lost notifications), scalable to 10B/day, low latency for urgent alerts (<1 second), delivery analytics

Estimate

# 10B notifications/day
# 10B / 86400 ~ 115K notifications/sec (peak ~350K/sec)
# Channel split: push 60%, email 25%, SMS 5%, in-app 10%

Architecture

The system follows a pipeline pattern: trigger services emit notification requests, which flow through validation, preference checks, template rendering, priority queuing, and finally channel-specific dispatchers.

# Notification flow:
# 1. Trigger service sends request:
#    { user_id: 42, type: "order_shipped",
#      data: { order_id: "ORD-123" } }
#
# 2. Notification Service:
#    a. Dedup: check idempotency key (was this already sent?)
#    b. Preferences: does user want "order_shipped" on push+email?
#    c. Template: render "order_shipped" with data
#       -> "Your order ORD-123 has been shipped!"
#    d. Rate limit: not too many notifications in short window
#
# 3. Priority Queue (Kafka):
#    URGENT (payment alerts) -> processed first
#    NORMAL (social) -> standard processing
#    LOW (marketing) -> best-effort, batched
#
# 4. Channel Dispatchers:
#    Push -> APNs (iOS) + FCM (Android)
#    Email -> SES / SendGrid
#    SMS -> Twilio
#    In-app -> WebSocket push

Key Components

  • Notification Service: Central orchestrator that validates, deduplicates, checks preferences, and renders templates
  • Template Engine: Stores notification templates with placeholders; renders with per-notification data
  • Preference Store: Per-user, per-notification-type, per-channel settings (Redis for fast lookups)
  • Priority Queues (Kafka): Separate topics for urgent, normal, and low priority
  • Channel Dispatchers: One per channel, handles provider-specific formatting and retries
  • Analytics Store: Tracks delivery, open, and click-through rates

Reliability Patterns

  • At-least-once delivery: Retry failed sends with exponential backoff; idempotency keys prevent duplicates at the provider level
  • Dead letter queue: Notifications that fail after max retries go to a DLQ for investigation
  • Circuit breaker: If a provider (e.g., APNs) is down, stop sending and queue messages until recovery
  • Fallback channels: If push fails, fall back to email or SMS for critical notifications

Design Considerations

  • User fatigue: Rate limit non-urgent notifications (max 5 marketing notifications per day per user)
  • Batching: Low-priority notifications can be batched into digests (daily summary email)
  • Timezone awareness: Do not send marketing notifications at 3 AM user-local time
  • Token management: Device push tokens expire and change; maintain a registry of active tokens per user
