Week 9 Exercises: Social and Messaging Systems

These exercises explore the fan-out problem, real-time messaging, and notification delivery covered this week.

Exercise 1: Celebrity Fan-Out Threshold Analysis

The hybrid fan-out model uses a follower threshold to decide between push and pull. But what is the optimal threshold?

  • Model a social network with 1M users following a power-law distribution (most users have few followers, a few have millions)
  • Simulate tweet creation with push-only, pull-only, and hybrid strategies
  • Vary the celebrity threshold from 1K to 100K followers

Tasks:

  • For each threshold, measure total cache writes (push cost) and DB reads at timeline load time (pull cost)
  • Plot total system cost vs threshold to find the sweet spot
  • What happens to timeline latency (p99) at each threshold?
  • Consider: should the threshold be static or dynamic based on current system load?
# Simplified cost model:
# push_cost = sum(followers) for all tweets by non-celebrities
# pull_cost = num_celebrities_followed * timeline_loads
# total_cost = push_cost + pull_cost
# Find threshold that minimizes total_cost

Exercise 2: Chat Delivery Guarantees

Design the message delivery guarantee system for a WhatsApp-like chat, handling these failure scenarios:

  • Sender's device loses connection mid-send
  • Chat server crashes after receiving message but before routing
  • Recipient's device is offline for 3 days then reconnects
  • Recipient has two devices (phone and desktop) -- both must receive the message

Tasks:

  • Design the message state machine: what states can a message be in, and what transitions are possible?
  • How do you implement exactly-once delivery (or effectively so) with at-least-once transport?
  • Draw the sequence diagram for a message that goes through all three receipt states (sent, delivered, read)
  • How does the offline message queue handle a user with 10,000 unread messages on reconnect?

Exercise 3: Notification System Load Test

Design a load test that simulates a flash sale notification being sent to 50M users simultaneously.

# Scenario: E-commerce flash sale starts at 12:00 PM
# Send push + email to 50M opted-in users within 60 seconds
# 50M / 60 = 833K notifications/sec across two channels

# Questions to address:
# 1. How many Kafka partitions do you need for the queue?
# 2. How many push dispatcher workers (APNs rate limit:
#    ~50K/sec per connection)?
# 3. What happens if SendGrid rate-limits your email sends?
# 4. How do you ensure the notification arrives BEFORE 12:00 PM
#    (not after the sale starts)?

Tasks:

  • Calculate the infrastructure needed: Kafka partitions, dispatcher workers, provider connections
  • Design the pre-warming strategy: prepare notification payloads in advance, send on trigger
  • How do you monitor delivery in real-time and detect if a channel is falling behind?
