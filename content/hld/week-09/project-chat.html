Project: Chat System Prototype

This project implements an in-memory chat server simulating WhatsApp-like messaging with WebSocket connection management, multi-server message routing, delivery receipts, group messaging, presence tracking, and offline message queuing.

Core Data Structures

from enum import Enum
from dataclasses import dataclass, field
from typing import Dict, List, Optional, Set
from collections import defaultdict, deque

class MessageStatus(Enum):
    SENT = "sent"           # Server received
    DELIVERED = "delivered"  # Recipient's device received
    READ = "read"           # Recipient opened chat

class PresenceState(Enum):
    ONLINE = "online"
    OFFLINE = "offline"
    TYPING = "typing"

@dataclass
class Message:
    msg_id: int
    conversation_id: str
    sender_id: int
    content: str
    timestamp: float
    status: MessageStatus = MessageStatus.SENT
    delivered_at: Optional[float] = None
    read_at: Optional[float] = None
    is_group: bool = False

@dataclass
class Conversation:
    conversation_id: str
    participants: Set[int]
    is_group: bool = False
    group_name: str = ""
    messages: List[Message] = field(default_factory=list)

Chat Server Simulation

Each chat server handles WebSocket connections for a subset of users. In production, each server would hold approximately 1M concurrent connections.

class ChatServer:
    def __init__(self, server_id: str):
        self.server_id = server_id
        self.connections: Dict[int, bool] = {}
        self.pending_messages: Dict[int, deque] = defaultdict(deque)

    def connect(self, user_id: int):
        self.connections[user_id] = True

    def disconnect(self, user_id: int):
        self.connections.pop(user_id, None)

    def is_connected(self, user_id: int) -> bool:
        return user_id in self.connections

    def deliver_message(self, user_id: int, message: Message) -> bool:
        if self.is_connected(user_id):
            self.total_messages_routed += 1
            return True  # delivered via WebSocket
        # Queue for later delivery
        self.pending_messages[user_id].append(message)
        return False

Message Routing

  • Session lookup: Redis maps each user_id to their current chat server
  • Same server: If sender and recipient are on the same server, deliver directly
  • Cross-server: Route message from sender's server to recipient's server via internal messaging
  • Offline: If recipient is offline, queue the message. Deliver all queued messages on reconnect.

Delivery Receipts

  • Sent: Acknowledged when the server receives and persists the message
  • Delivered: Acknowledged when the recipient's device receives the message (even if not opened)
  • Read: Acknowledged when the recipient opens the conversation containing the message

Group Messaging

When a message is sent to a group, the server fans it out to all participants. Each participant gets their own delivery receipt tracking. The message content is stored once; each participant has a reference to it.

Presence Tracking

  • Heartbeat-based: clients send periodic heartbeats to indicate they are online
  • If no heartbeat within 30 seconds, mark as offline and record last-seen timestamp
  • Typing indicators are transient events, not persisted

Key Takeaways

  • The session service (user-to-server mapping) is the critical lookup that enables message routing
  • Offline queuing ensures no messages are lost even when recipients are disconnected for days
  • Delivery receipts require bidirectional acknowledgment flows, which adds complexity but is essential for user trust
