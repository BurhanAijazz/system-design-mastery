YouTube / Video Platform Design

YouTube is the heaviest system design problem: petabytes of video content, millions of concurrent viewers, and a complex transcoding pipeline. The key challenge is handling massive storage and bandwidth at global scale.

Define: Requirements

  • Functional: Upload videos (any format, up to 256 GB), stream with adaptive quality, search videos, personalized recommendations, like/comment/subscribe, view count tracking
  • Non-Functional: Resumable uploads for large files, playback starts within 2 seconds, 99.99% streaming availability, eventual consistency for view counts and comments
  • Out of Scope: Live streaming, monetization/ads, Shorts

Estimate

# 2B MAU, 500M DAU
# 500 hours of video uploaded per minute (YouTube's real number)

# UPLOAD:
#   500 hrs/min x 60 x 24 = 720K hours/day
#   Transcoded to 3 resolutions: ~144 TB/day new storage

# STREAMING:
#   500M videos/day x 10 min avg x 2.5 MB/min = 12.5 PB/day
#   12.5 PB / 86400 = 145 GB/sec sustained
#   Peak (3x): 435 GB/sec -> CDN is critical

# STORAGE PER YEAR:
#   144 TB/day x 365 = 52.5 PB/year
#   After 10 years: 500+ PB (half an exabyte!)
#   S3-compatible object storage is the ONLY option

# METADATA: 500M videos x 10 KB = 5 TB (manageable)

Architecture

# High-level flow:
# Client -> API Gateway -> Upload Service (chunked upload)
#                                |
#                          Raw Storage (S3)
#                                |
#                     Transcoding Pipeline (parallel)
#                       /      |        \
#                    240p    720p     1080p    4K
#                       \      |        /
#                     HLS Manifest Generator
#                                |
#                           CDN (edge)
#                                |
#                          Client streams

Upload Pipeline

  • Chunked upload: Break large files into 5 MB chunks for resumable upload. Client tracks which chunks are uploaded.
  • Pre-signed URLs: Upload directly to S3, bypassing application servers for throughput.
  • Transcoding: Parallel workers transcode to multiple resolutions (240p, 360p, 480p, 720p, 1080p, 4K). Each resolution has different bitrate and processing time.
  • HLS/DASH manifests: Generate manifest files listing available resolutions and segments for adaptive bitrate streaming.

Streaming and Adaptive Bitrate

  • Video is split into 2-10 second segments, each available in multiple resolutions
  • Client requests segments based on available bandwidth (adaptive bitrate)
  • CDN caches popular segments at edge locations globally
  • Less popular (long-tail) content may require origin fetch from S3

View Counting

  • Real-time counting would create a hot partition (viral videos get millions of views/minute)
  • Solution: count locally on each server, batch-aggregate periodically (every 30 seconds)
  • View counts are eventually consistent -- acceptable for this use case

Key Design Decisions

  • CDN is the product: At 435 GB/sec peak, serving from origin is impossible. CDN handles 95%+ of traffic.
  • Transcoding priority: Lower resolutions first (users can start watching sooner), then higher resolutions in background.
  • Content moderation: Automated scanning during transcoding phase, before content is published.
  • Thumbnail generation: Auto-extract key frames during transcoding; also allow custom uploads.
