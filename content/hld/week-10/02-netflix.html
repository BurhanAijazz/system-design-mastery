Netflix / Streaming Service Design

Netflix differs fundamentally from YouTube: it has a curated catalog of ~15,000 titles (not billions of user-uploaded videos), a subscription model, and content is pre-positioned at ISP-level CDN nodes. The challenge is bandwidth, not storage.

Define: Requirements

  • Functional: Browse catalog, stream video with adaptive quality, personalized recommendations, user profiles, continue watching, download for offline, parental controls
  • Non-Functional: Playback within 2 seconds, zero buffering (adaptive quality over stalling), 99.99% availability globally, 200M+ concurrent streams

How Netflix Differs from YouTube

  • YouTube: User-generated, billions of videos, long tail distribution, upload pipeline critical, ad-supported
  • Netflix: Curated catalog (~15K titles), content pre-transcoded hours before release, every title gets many views, subscription model, content pre-positioned at ISP CDN nodes

Estimate

# 250M subscribers, 100M DAU
# Average session: 2 hours

# CONCURRENT STREAMS:
#   100M / 24 x 2 hrs avg = ~8M concurrent average
#   Peak (7-11 PM): ~20M concurrent
#   20M x 5 Mbps avg = 100 Tbps peak bandwidth!

# STORAGE:
#   15,000 titles x 100 GB (all resolutions + codecs + languages)
#   Total: 1.5 PB catalog
#   Small compared to YouTube! Challenge is BANDWIDTH, not storage.

# METADATA:
#   15K titles x 50 KB = 750 MB (tiny!)
#   250M user profiles x 5 KB = 1.25 TB
#   Watch history: 250M x 200 entries x 100B = 5 TB

Open Connect: Netflix's Custom CDN

Netflix built its own CDN called Open Connect because commercial CDNs cannot handle 100+ Tbps cost-effectively:

  • ISP-embedded appliances: Custom hardware boxes placed inside ISP data centers
  • Pre-positioned content: Popular titles pushed to ISP nodes before prime time
  • 95%+ traffic served from ISP edge: Traffic never reaches Netflix's origin servers
  • Cost savings: Reduces bandwidth cost by 10-100x vs commercial CDN

Architecture

  • Control Plane (AWS): API, authentication, recommendation engine, user profiles, billing -- all non-video services run on AWS microservices
  • Data Plane (Open Connect): Video streaming traffic flows entirely through ISP-embedded CDN nodes
  • Content Pipeline: Ingest master copy, transcode to 10+ resolution/codec combinations, distribute to Open Connect nodes worldwide

Recommendation Engine

  • The home screen IS the product -- users decide what to watch based on recommendations
  • Pre-computed offline (batch ML pipeline), personalized per profile
  • A/B testing is core to Netflix culture -- every UI change is tested on user segments

Key Design Decisions

  • Pre-transcode everything: Unlike YouTube (on-demand transcoding), Netflix transcodes all content ahead of time in every resolution and codec combination
  • Adaptive bitrate: Client dynamically switches quality based on bandwidth; prioritizes smooth playback over resolution
  • Microservices architecture: Netflix pioneered microservices at scale (800+ services); each independently deployable and scalable
  • Chaos engineering: Netflix invented Chaos Monkey to randomly kill services and verify resilience
