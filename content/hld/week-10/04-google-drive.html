Google Drive / File Storage Design

Google Drive is about file sync, delta sync, block-level deduplication, conflict resolution, and versioning. The core challenge is syncing files across devices efficiently -- only transferring the parts that changed.

Define: Requirements

  • Functional: Upload/download files, sync across devices (desktop/mobile/web), share files with permissions, version history with restore, real-time collaborative editing, search across files, offline access
  • Non-Functional: Sync within seconds of file change, never lose data (durability is paramount), handle files from 1 KB to 5 TB, minimize bandwidth (only sync changes), millions of concurrent syncs
  • Out of Scope: Google Docs real-time editing (OT/CRDT), Sheets, Slides, admin console

Estimate

# 1B registered users, 50M DAU
# 10 file operations/day per active user, avg file: 500 KB

# OPERATIONS:
#   50M x 10 = 500M ops/day = ~5,800 QPS (peak ~17K)

# STORAGE:
#   1B users x 5 GB avg used = 5 EB (exabytes!)
#   Daily new: 500M x 500KB x 20% new = 50 TB/day
#   With delta sync: only ~20% changed = 10 TB/day
#   With dedup: ~30% savings = 7 TB/day actual new storage

# METADATA:
#   50B files total x 1 KB = 50 TB
#   PostgreSQL cluster or Spanner for strong consistency

Architecture

# Desktop sync daemon detects file changes
# -> Notification Service (WebSocket for push sync)
# -> Block Sync Service
#    1. Split file into blocks (4 MB chunks)
#    2. Hash each block (SHA-256)
#    3. Compare hashes with server
#    4. Upload ONLY changed blocks (delta sync)
#    5. Server deduplicates blocks across all users
# -> Block Store (S3): content-addressable storage
# -> Metadata DB: file tree, permissions, versions

Block-Level Delta Sync

This is the key innovation that makes Drive efficient:

  • Chunking: Files are split into fixed or content-defined blocks (typically 4 MB)
  • Hashing: Each block gets a SHA-256 hash (content-addressable)
  • Delta detection: Only blocks with changed hashes are uploaded
  • Deduplication: If two users upload the same file, blocks are stored once, referenced twice
# Example: 100 MB file, 25 blocks of 4 MB each
# User edits 1 paragraph -> only 1 block changes
# Upload: 4 MB instead of 100 MB = 96% bandwidth saving

# With dedup across users:
# 1000 users share the same PDF -> stored once
# Reference count tracks how many users point to each block

Conflict Resolution

  • Problem: Two devices edit the same file while offline, then both sync
  • Detection: Compare version numbers -- if both edits are based on the same version, it is a conflict
  • Resolution: Keep both versions, rename the later one as "filename (conflict copy)", let the user merge manually
  • Prevention: Real-time sync notifications reduce the window for conflicts

Version History

  • Every file change creates a new version (list of block hashes)
  • Restore any previous version by reassembling its blocks
  • Storage-efficient: versions share unchanged blocks through deduplication
  • Retention policy: keep all versions for 30 days, then compress to major versions only

Key Design Decisions

  • Block storage is the foundation: Everything (sync, dedup, versioning) builds on content-addressable blocks
  • Notification-driven sync: Push notifications (WebSocket) tell clients when to pull changes, avoiding polling
  • Strong consistency for metadata: File tree and permissions use Spanner or PostgreSQL with synchronous replication
  • Eventual consistency for content: Block uploads can be slightly delayed without affecting user experience
