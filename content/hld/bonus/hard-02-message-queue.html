Design: Distributed Message Queue

Design a Kafka-like distributed message queue for high-throughput event streaming.

Requirements

  • Durable, ordered message delivery within a partition
  • Consumer groups with partition assignment for parallel consumption
  • At-least-once delivery with consumer offset tracking
  • Replication for fault tolerance; survive broker failures

High-Level Architecture

  • Producers — Publish messages to a topic, optionally keyed for partition routing
  • Brokers — Store partitions as append-only segment files on disk
  • Controller — Manages partition leader election and broker membership
  • Consumer Groups — Each partition assigned to exactly one consumer in a group
  • ZooKeeper / KRaft — Metadata coordination and leader election

Key Components

from collections import defaultdict

class Partition:
    def __init__(self, topic, partition_id):
        self.topic = topic
        self.id = partition_id
        self.log = []  # append-only message log
        self.replicas = []

    def append(self, message):
        offset = len(self.log)
        self.log.append({"offset": offset, "data": message})
        self._replicate({"offset": offset, "data": message})
        return offset

    def read(self, offset, max_count=10):
        return self.log[offset:offset + max_count]

    def _replicate(self, entry):
        for replica in self.replicas:
            replica.append_replica(entry)

class ConsumerGroup:
    def __init__(self, group_id):
        self.group_id = group_id
        self.offsets = {}  # (topic, partition) -> committed offset
        self.assignments = {}  # consumer_id -> [partitions]

    def commit(self, topic, partition, offset):
        self.offsets[(topic, partition)] = offset

    def poll(self, consumer_id, broker):
        messages = []
        for p in self.assignments.get(consumer_id, []):
            offset = self.offsets.get((p.topic, p.id), 0)
            batch = p.read(offset, max_count=100)
            messages.extend(batch)
        return messages

Scaling Considerations

  • More partitions = more parallelism, but also more rebalance overhead
  • ISR (in-sync replicas): only acknowledge when min ISR replicas have the message
  • Log compaction retains only latest value per key for changelog topics
  • Consumer lag monitoring triggers auto-scaling of consumer instances
