Project: Event-Driven Order Pipeline

Simulate a complete order processing pipeline with events, retry logic, dead letter queue, and idempotency protection. This project builds intuition about event-driven systems and their failure modes.

Pipeline Architecture

Full order flow: Order Created -> Payment Processing -> Inventory Reserved -> Notification Sent -> Completed. Each step can fail and trigger compensating actions.

Data Models

from dataclasses import dataclass, field
from typing import List, Dict, Set
from enum import Enum
import hashlib
import time

class OrderStatus(Enum):
    CREATED = "created"
    PAYMENT_PROCESSING = "payment_processing"
    PAYMENT_COMPLETED = "payment_completed"
    PAYMENT_FAILED = "payment_failed"
    INVENTORY_RESERVED = "inventory_reserved"
    INVENTORY_FAILED = "inventory_failed"
    COMPLETED = "completed"
    FAILED = "failed"
    COMPENSATED = "compensated"

@dataclass
class Order:
    id: str
    user_id: str
    items: List[str]
    amount: float
    status: OrderStatus = OrderStatus.CREATED
    idempotency_key: str = ""
    attempts: int = 0

    def __post_init__(self):
        if not self.idempotency_key:
            self.idempotency_key = hashlib.md5(
                f"{self.id}:{self.user_id}:{self.amount}".encode()
            ).hexdigest()[:12]

@dataclass
class Event:
    event_type: str
    order_id: str
    data: Dict
    source: str
    timestamp: float = 0.0

Pipeline Features

  • Configurable failure rates: Set failure probability for each service (payment: 10%, inventory: 5%)
  • Retry with exponential backoff: Failed steps are retried with increasing delays
  • Dead letter queue: Messages that fail after max retries are moved to a DLQ for manual review
  • Idempotency protection: Duplicate events are detected and ignored using idempotency keys
  • Compensation on failure: If payment fails after inventory is reserved, inventory is released
  • Event log and audit trail: Every event is logged for debugging and replay

Pipeline Processor Skeleton

class OrderPipeline:
    """Event-driven order processing with retry and DLQ."""

    def __init__(self, payment_failure_rate=0.1, inventory_failure_rate=0.05):
        self.orders: Dict[str, Order] = {}
        self.event_log: List[Event] = []
        self.dead_letter_queue: List[Event] = []
        self.processed_keys: Set[str] = set()
        self.max_retries = 3

    def process_order(self, order: Order):
        """Run the order through the pipeline stages."""
        # Stage 1: Process payment
        if not self._process_payment(order):
            self._compensate(order)
            return

        # Stage 2: Reserve inventory
        if not self._reserve_inventory(order):
            self._refund_payment(order)
            self._compensate(order)
            return

        # Stage 3: Send notification
        self._send_notification(order)
        order.status = OrderStatus.COMPLETED

    def _compensate(self, order: Order):
        """Run compensating transactions in reverse order."""
        order.status = OrderStatus.COMPENSATED
        self.event_log.append(Event(
            "OrderCompensated", order.id,
            {"reason": order.status.value}, "pipeline"
        ))

Metrics to Track

  • Success rate: Percentage of orders that complete successfully
  • Average retries: How many retries per order on average
  • DLQ depth: Number of messages stuck in the dead letter queue
  • Compensation count: How often compensating transactions are triggered
  • End-to-end latency: Time from order creation to completion

Experiments

  • Run 1,000 orders with 10% payment failure rate. Observe retry behavior and DLQ depth
  • Increase failure rate to 50%. How does the pipeline behave under stress?
  • Send duplicate orders with the same idempotency key. Verify they are processed only once
