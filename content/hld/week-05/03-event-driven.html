Event-Driven Architecture

Events as the backbone of modern distributed systems. Instead of services calling each other directly, services publish events and other services react. This creates loose coupling and extensibility.

Imperative vs Event-Driven

  • Imperative: OrderService calls PaymentService, then InventoryService, then NotificationService directly
  • Event-Driven: OrderService publishes "OrderCreated". Payment, Inventory, Notification, and Analytics services all subscribe independently

Event Types

  • Domain Events: Something happened -- "OrderPlaced", "PaymentProcessed". Past tense, immutable facts
  • Integration Events: Cross-service communication via message broker. Loose coupling
  • Commands: Requests to do something -- "ProcessPayment", "SendEmail". Can fail and be rejected

Simple Event Bus

from dataclasses import dataclass, field
from typing import Dict, Any, List, Callable
from collections import defaultdict
import time

@dataclass
class Event:
    event_type: str
    data: Dict[str, Any]
    event_id: int = 0
    timestamp: float = 0.0
    source: str = ""

    def __post_init__(self):
        if not self.timestamp:
            self.timestamp = time.time()

class EventBus:
    """In-process publish-subscribe event bus."""
    def __init__(self):
        self.subscribers: Dict[str, List[Callable]] = defaultdict(list)
        self.event_log: List[Event] = []

    def subscribe(self, event_type: str, handler: Callable):
        self.subscribers[event_type].append(handler)

    def publish(self, event: Event):
        self.event_log.append(event)
        for handler in self.subscribers.get(event.event_type, []):
            handler(event)

# Usage:
bus = EventBus()
bus.subscribe("OrderCreated", lambda e: print(f"Payment for {e.data}"))
bus.subscribe("OrderCreated", lambda e: print(f"Inventory for {e.data}"))
bus.publish(Event("OrderCreated", {"order_id": "123", "amount": 99.99}))

Choreography vs Orchestration

  • Choreography: Each service listens for events and decides what to do. No central coordinator. Good for simple flows but hard to debug complex ones
  • Orchestration: A central orchestrator tells each service what to do and when. Easier to understand and monitor. Single point of failure risk

Event Sourcing

Instead of storing current state, store the sequence of events that led to the current state. The current state is derived by replaying events.

  • Pro: Complete audit trail, can reconstruct state at any point in time
  • Pro: Events are immutable -- no data loss from overwrites
  • Con: More complex queries (need to rebuild state from events)
  • Con: Event schema evolution is challenging

CQRS: Command Query Responsibility Segregation

Separate the write model (commands) from the read model (queries). Write to an event store, build optimized read views from the events.

  • Write side: append events to the event store
  • Read side: materialized views optimized for specific query patterns
  • Often paired with event sourcing for maximum flexibility
