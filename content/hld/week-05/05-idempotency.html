Idempotency and Exactly-Once Delivery

The hardest problem in distributed messaging: processing a message exactly once. The practical solution is at-least-once delivery + idempotent processing.

Message Delivery Guarantees

  • At-Most-Once: Fire and forget. If the network drops it, the message is lost. OK for metrics and logs
  • At-Least-Once: Retry until acknowledged. Message arrives but may arrive multiple times. The default for Kafka, RabbitMQ, SQS
  • Exactly-Once: Each message processed exactly one time. Extremely hard due to the Two Generals' Problem

The Two Generals' Problem proves there is no protocol that guarantees exactly-once delivery over an unreliable network. The practical solution: at-least-once delivery + idempotent consumers.

What Is Idempotency?

An operation is idempotent if performing it multiple times has the same effect as performing it once. f(f(x)) = f(x).

# Naturally idempotent operations:
# SET balance = 500        (same result every time)
# DELETE FROM orders WHERE id = 123  (deleting twice = same as once)
# PUT /users/123 {"name": "Alice"}   (full replacement)

# NOT idempotent:
# balance += 100           (each call adds more)
# INSERT INTO orders (...)  (each call creates a new row)
# POST /users {"name": "Alice"}  (each call creates a new user)

Idempotency Key Pattern

import hashlib
from typing import Dict, Set

class IdempotentProcessor:
    """Process messages exactly once using idempotency keys."""

    def __init__(self):
        self.processed_keys: Set[str] = set()
        self.results: Dict[str, dict] = {}

    def process(self, idempotency_key: str, message: dict) -> dict:
        # Check if already processed
        if idempotency_key in self.processed_keys:
            return self.results[idempotency_key]  # Return cached result

        # Process the message
        result = self._do_work(message)

        # Record that we processed it
        self.processed_keys.add(idempotency_key)
        self.results[idempotency_key] = result
        return result

    def _do_work(self, message: dict) -> dict:
        return {"status": "processed", "order_id": message.get("order_id")}

# Usage: client generates idempotency key
key = hashlib.md5(f"order:123:user:456:amount:99.99".encode()).hexdigest()
processor = IdempotentProcessor()
result1 = processor.process(key, {"order_id": "123", "amount": 99.99})
result2 = processor.process(key, {"order_id": "123", "amount": 99.99})
# result1 == result2, work only done once

Implementation Strategies

  • Database unique constraint: INSERT with a unique idempotency_key column. Duplicates fail with a constraint violation
  • Redis SET NX: SETNX idempotency:{key} 1 EX 86400. Atomic check-and-set with TTL
  • Kafka consumer offsets: Commit offset only after successful processing. On restart, replay from last committed offset

Stripe's Idempotency Model

Stripe's API is the gold standard. Pass an Idempotency-Key header with every POST request. Stripe caches the response for 24 hours. Retrying with the same key returns the same response without re-processing.

Dead Letter Queue (DLQ)

Messages that fail repeatedly (after N retries) are moved to a Dead Letter Queue for manual inspection instead of blocking the pipeline.
