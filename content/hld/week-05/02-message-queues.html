Message Queues: Kafka, RabbitMQ, SQS

Three dominant messaging technologies, each optimized for different patterns. Understanding when to use which is a critical interview skill.

Message Queue vs Event Stream

Model 1: Message Queue (Point-to-Point)

Each message is delivered to exactly one consumer. After processing, the message is deleted from the queue.

  • Examples: RabbitMQ, Amazon SQS, ActiveMQ
  • Use for: task queues, job processing, work distribution

Model 2: Event Stream (Pub-Sub / Log-Based)

Messages are retained in a log. Each consumer group gets every message independently. Consumers track their position (offset).

  • Examples: Apache Kafka, Amazon Kinesis, Redpanda
  • Use for: event streaming, CDC, audit logs, real-time analytics

RabbitMQ: Exchanges, Queues, and Routing

# RabbitMQ architecture:
# Producer -> Exchange -> Queue -> Consumer
# Exchange types determine routing logic:

# DIRECT: route by exact routing key
# "order.created" -> order_queue

# FANOUT: broadcast to ALL bound queues
# event -> notification_queue, analytics_queue, audit_queue

# TOPIC: route by pattern matching
# "order.created" matches "order.*"
# "payment.failed" matches "payment.*"

class SimulatedExchange:
    """Simplified RabbitMQ exchange simulation."""
    def __init__(self, exchange_type="direct"):
        self.exchange_type = exchange_type
        self.bindings = {}  # routing_key -> [queues]

    def bind(self, queue, routing_key):
        self.bindings.setdefault(routing_key, []).append(queue)

    def publish(self, message, routing_key):
        if self.exchange_type == "fanout":
            for queues in self.bindings.values():
                for q in queues:
                    q.append(message)
        elif self.exchange_type == "direct":
            for q in self.bindings.get(routing_key, []):
                q.append(message)

Apache Kafka: Distributed Event Log

  • Topics: logical channels for events (e.g., "orders", "payments")
  • Partitions: each topic is split into partitions for parallelism
  • Consumer groups: each group reads every message; within a group, partitions are divided among consumers
  • Retention: messages kept for days/weeks (default 7 days), not deleted after consumption
  • Ordering: guaranteed within a partition, not across partitions

Amazon SQS: Managed Queue

  • Fully managed, no infrastructure to maintain
  • Standard: at-least-once delivery, best-effort ordering, nearly unlimited throughput
  • FIFO: exactly-once processing, strict ordering, 3,000 msg/sec
  • Visibility timeout: message hidden from other consumers while being processed

Choosing the Right Technology

  • RabbitMQ: complex routing, priority queues, request-reply patterns. Best for task distribution
  • Kafka: high throughput, event replay, multiple consumers. Best for event streaming and analytics pipelines
  • SQS: zero ops, simple queue semantics, auto-scaling. Best for decoupling AWS services
