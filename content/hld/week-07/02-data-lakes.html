Data Lakes vs Data Warehouses

Where do you put your data once it outgrows a single database? Two major paradigms: warehouses (structured, schema-on-write) vs lakes (raw, schema-on-read).

Data Warehouse

A data warehouse stores structured data optimized for analytical queries. Examples: Snowflake, BigQuery, Redshift, Azure Synapse.

  • Schema-on-write: Define the schema before loading data
  • Structured: Tables with columns, types, constraints
  • SQL interface: Analysts query with standard SQL
  • Optimized for read-heavy analytical queries (OLAP), not transactional workloads

Star Schema

The most common warehouse design. A central fact table surrounded by dimension tables.

from dataclasses import dataclass
from typing import Dict, List
from collections import defaultdict

@dataclass
class FactSale:
    """Fact table: measurements/events (large, billions of rows)."""
    date_id: int
    product_id: int
    store_id: int
    customer_id: int
    quantity: int
    revenue: float

@dataclass
class DimProduct:
    """Dimension table: context (small, thousands of rows)."""
    product_id: int
    name: str
    category: str
    brand: str

# GROUP BY d.category

Data Lake

A data lake stores raw data in its native format. Schema is applied when reading, not when writing.

  • Schema-on-read: Store first, structure later
  • Any format: JSON, CSV, Parquet, Avro, images, logs
  • Cheap storage: Built on object storage (S3, GCS) -- pennies per GB
  • Flexible: Raw data preserved, can re-process as needs change
  • Risk: Without governance, becomes a "data swamp" -- raw data nobody can use

Data Lakehouse

Modern approach that combines lake and warehouse features. Store raw data on cheap object storage but add warehouse-like structure (ACID transactions, schema enforcement, indexing).

  • Examples: Databricks Delta Lake, Apache Iceberg, Apache Hudi
  • One copy of data serves both raw analytics and structured BI queries

Warehouse vs Lake vs Lakehouse

  • Warehouse: Best for BI/reporting, structured analytics, SQL-native teams
  • Lake: Best for ML/data science, raw data exploration, cost-sensitive storage
  • Lakehouse: Best when you need both, want one copy of data, modern stack

Column-Oriented Storage

Warehouses store data by column, not by row. This makes analytical queries (aggregations over a few columns across millions of rows) dramatically faster because only the needed columns are read from disk.

# Row-oriented (OLTP): store each row contiguously
# Row 1: [Alice, 30, NYC, $50]
# Row 2: [Bob,   25, LA,  $30]
# Query "SELECT name" reads ALL columns for every row

# Column-oriented (OLAP): store each column contiguously
# Name column:   [Alice, Bob, Carol, ...]
# Amount column: [50, 30, 100, ...]
# Query "SELECT SUM(amount)" reads ONLY the amount column
# 10x-100x faster for analytical queries
