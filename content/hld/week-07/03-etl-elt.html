ETL/ELT Pipeline Design

How data moves from source systems to analytics destinations. ETL and ELT are the plumbing of every data-driven company.

ETL vs ELT

  • ETL (Extract, Transform, Load): Transform before loading. Traditional approach (Informatica, Talend). Only clean data enters the warehouse. Slow iteration
  • ELT (Extract, Load, Transform): Load raw data first, transform inside the warehouse. Modern approach (dbt, Spark). Raw data preserved, faster iteration
from typing import List, Dict, Any
from collections import defaultdict
import random

# --- EXTRACT: read from source systems ---

def extract_orders(num_orders: int) -> List[Dict]:
    """Simulate extracting orders from an operational database."""
    rng = random.Random(42)
    statuses = ["completed", "pending", "cancelled", "refunded"]
    orders = []
    for i in range(num_orders):
        orders.append({
            "order_id": f"ORD-{i:06d}",
            "user_id": f"USR-{rng.randint(0, 999):04d}",
            "amount": round(rng.uniform(5, 500), 2),
            "status": rng.choice(statuses),
            "created_at": f"2024-01-{rng.randint(1,31):02d}",
            "raw_items": f"item_{rng.randint(1,100)},item_{rng.randint(1,100)}",
        })
    return orders

# --- TRANSFORM: clean, validate, enrich ---

def transform_orders(raw_orders: List[Dict]) -> List[Dict]:
    """Clean and structure raw order data."""
    transformed = []
    for order in raw_orders:
        if order["status"] == "cancelled":
            continue  # Filter out cancelled orders
        transformed.append({
            "order_id": order["order_id"],
            "user_id": order["user_id"],
            "amount": float(order["amount"]),
            "status": order["status"].upper(),
            "items": order["raw_items"].split(","),
            "item_count": len(order["raw_items"].split(",")),
        })
    return transformed

# --- LOAD: write to destination ---

def load_to_warehouse(orders: List[Dict], warehouse: Dict):
    """Load transformed data into the warehouse."""
    for order in orders:
        warehouse[order["order_id"]] = order

Pipeline Components

  • Extract: Read from source systems (databases, APIs, files, streams). Techniques: full load, incremental (CDC), log-based replication
  • Transform: Clean (remove nulls, fix types), validate (business rules), enrich (join with reference data), aggregate (pre-compute metrics)
  • Load: Write to destination (warehouse, lake, search index). Modes: full replace, append, upsert (merge)

Change Data Capture (CDC)

Instead of extracting all data every time, capture only what changed since the last extract. CDC reads the database's transaction log to stream inserts, updates, and deletes in real-time.

  • Tools: Debezium, AWS DMS, Fivetran
  • Enables near-real-time data pipelines instead of nightly batch jobs

Modern Data Stack

  • Extraction: Fivetran, Airbyte (managed connectors)
  • Storage: Snowflake, BigQuery, Redshift (cloud warehouse)
  • Transformation: dbt (SQL-based transforms in the warehouse)
  • Orchestration: Airflow, Dagster, Prefect (schedule and monitor pipelines)
  • Visualization: Looker, Tableau, Metabase (dashboards and reports)

Pipeline Reliability

  • Idempotency: Running the pipeline twice produces the same result (no duplicates)
  • Data quality checks: Validate row counts, null rates, and schema after each stage
  • Alerting: Notify on pipeline failures, data quality violations, SLA breaches
  • Backfill: Ability to reprocess historical data when logic changes
