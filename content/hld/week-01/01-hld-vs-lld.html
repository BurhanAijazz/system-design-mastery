High-Level Design vs Low-Level Design

System design operates at two complementary levels. High-Level Design (HLD) focuses on the architecture of the entire system — its components, how they communicate, and how the system scales. Low-Level Design (LLD) focuses on the internal structure of individual components — classes, interfaces, data structures, and algorithms.

What Is High-Level Design?

HLD answers the question: "How do we build a system that serves millions of users reliably?" It deals with:

  • System architecture and component decomposition
  • Service boundaries and communication protocols
  • Data storage strategies (SQL vs NoSQL, sharding, replication)
  • Scalability (horizontal vs vertical scaling)
  • Load balancing, caching, and CDN placement
  • Availability, consistency, and partition tolerance trade-offs (CAP theorem)
  • Capacity estimation and bottleneck identification

What Is Low-Level Design?

LLD answers the question: "How do we structure the code inside a single service so it is maintainable, testable, and extensible?" It deals with:

  • Class diagrams and object relationships
  • Design patterns (Strategy, Observer, Factory, etc.)
  • SOLID principles and clean architecture
  • API contract design for individual services
  • Data structures and algorithm choices
  • Error handling and edge cases

# Low-Level Design: URL Shortener Key Generation Service
#
# This is ONE component from the HLD above, designed in detail.

from abc import ABC, abstractmethod
import hashlib
import string


class KeyGenerator(ABC):
    """Interface for short URL key generation strategies."""

    @abstractmethod
    def generate(self, long_url: str) -> str:
        pass


class Base62KeyGenerator(KeyGenerator):
    """Generates keys using Base62 encoding of a counter."""

    CHARS = string.ascii_letters + string.digits  # 62 characters

    def __init__(self, counter_service):
        self._counter = counter_service

    def generate(self, long_url: str) -> str:
        counter_value = self._counter.next()
        return self._encode(counter_value)

    def _encode(self, num: int) -> str:
        if num == 0:
            return self.CHARS[0]
        result = []
        while num > 0:
            result.append(self.CHARS[num % 62])
            num //= 62
        return "".join(reversed(result))


class MD5KeyGenerator(KeyGenerator):
    """Generates keys using MD5 hash truncation."""

    def generate(self, long_url: str) -> str:
        hash_hex = hashlib.md5(long_url.encode()).hexdigest()
        return hash_hex[:7]  # Take first 7 characters


# LLD decisions:
# - Which pattern? (Strategy for key generation)
# - Which data structure? (Base62 encoding)
# - Error handling? (collision detection)
# - Thread safety? (counter synchronization)

How HLD and LLD Work Together

# The relationship between HLD and LLD:
#
# HLD says: "We need a Key Generation Service"
# LLD says: "Here's how to implement it with Strategy pattern,
#            Base62 encoding, and collision handling"
#
# HLD says: "Use Redis for caching with a 24-hour TTL"
# LLD says: "Here's the CacheRepository interface, the Redis
#            implementation, and the cache invalidation logic"
#
# HLD says: "API servers behind a load balancer"
# LLD says: "Here's the API route handlers, request validation,
#            rate limiting middleware, and error response format"

When Each Matters in Interviews

  • HLD interviews (45-60 min): You are given a system to design from scratch (e.g., "Design Twitter"). You sketch components, discuss trade-offs, estimate capacity, and justify architectural choices.
  • LLD interviews (45-60 min): You are given a specific component to design in detail (e.g., "Design a parking lot system"). You define classes, interfaces, and patterns.
  • Real projects: You need both. HLD comes first to define the architecture, then LLD for each component.

Common Mistakes

  • Jumping to LLD too early: Designing class hierarchies before understanding the overall architecture leads to over-engineered components that do not fit together.
  • Staying too high-level: A system design with no detail on how individual components work internally is just a block diagram, not a design.
  • Ignoring trade-offs: Every HLD decision has trade-offs (latency vs consistency, cost vs redundancy). Acknowledge them explicitly.

The Design Process

# Recommended approach for any system design:
#
# 1. REQUIREMENTS (5 min)
#    - Functional: What does the system do?
#    - Non-functional: Scale, latency, availability targets
#
# 2. ESTIMATION (5 min)
#    - Users, requests/sec, storage, bandwidth
#
# 3. HIGH-LEVEL DESIGN (15-20 min)
#    - Component diagram
#    - Data flow
#    - API contracts between components
#
# 4. DEEP DIVE (15-20 min)
#    - Pick 1-2 critical components
#    - Low-level design of those components
#    - Discuss trade-offs and alternatives
#
# 5. SCALING & RELIABILITY (5-10 min)
#    - How does this handle 10x traffic?
#    - What happens when a component fails?

Key Takeaways

  • HLD is about the forest; LLD is about the trees. You need both.
  • Start with HLD to define system boundaries, then drill into LLD for critical components.
  • HLD decisions constrain LLD choices, and LLD feasibility can force HLD changes.
  • In interviews, explicitly state which level you are operating at and transition consciously.
