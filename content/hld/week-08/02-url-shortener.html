URL Shortener Design

The URL shortener is the most classic HLD interview question. It covers key generation, caching, storage estimation, and a clean API design. This lesson walks through the full DERSC framework.

Define: Requirements

  • Functional: Generate a short URL from a long URL; redirect short URL to original; optional custom aliases, expiration, and click analytics
  • Non-Functional: High availability for redirects, low latency (<50ms), short URLs should be as short as possible, not guessable
  • Out of Scope: User accounts, rate limiting, link previews

Estimate: Back-of-Envelope

# Assumptions:
#   100M new URLs/day, Read:Write = 10:1

# QPS:
#   Write: 100M / 86400 ~ 1,200 QPS (peak ~3,600)
#   Read:  1B / 86400 ~ 12,000 QPS (peak ~36,000)

# Storage (5-year horizon):
#   Per URL record: ~500 bytes
#   Daily: 100M x 500B = 50 GB/day
#   5 years: ~91 TB

# URL length with Base62:
#   62^7 = 3.5 TRILLION combinations
#   At 100M/day, lasts 95 years. 7 chars is sufficient.

Architect: API and Data Model

# API Design:
# POST /api/v1/urls
#   Request:  { "long_url": "https://...", "custom_alias": "my-link" }
#   Response: { "short_url": "https://short.ly/aB3x7Kp" }

# GET /{short_code} -> 301/302 redirect to long_url

# Key Generation - Base62 Encoding:
BASE62 = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"

def base62_encode(num: int) -> str:
    if num == 0:
        return BASE62[0]
    result = []
    while num > 0:
        result.append(BASE62[num % 62])
        num //= 62
    return ''.join(reversed(result))

Core Components

  • Key Generation Service: Counter-based with Base62 encoding or MD5/SHA hash truncation. Counter avoids collisions; hash approach is stateless.
  • Database: Simple key-value mapping (short_code -> long_url + metadata). NoSQL like DynamoDB or sharded MySQL.
  • Cache (Redis): LRU cache for hot URLs. 20% of daily reads fit in ~100 GB Redis instance.
  • Load Balancer: Distributes redirect traffic across read-optimized servers.

Scale: Caching and Sharding

  • Cache layer: Redis in front of DB. Cache hit avoids DB lookup entirely.
  • DB sharding: Partition by hash of short_code for even distribution.
  • 301 vs 302 redirect: 301 (permanent) reduces server load but loses analytics. 302 (temporary) ensures every click hits the server for tracking.
  • Expiration: Background cleanup job deletes expired URLs and evicts from cache.

Key Trade-offs

  • Counter-based keys are sequential (predictable) but collision-free. Hash-based keys are random but may collide.
  • Custom aliases require uniqueness checks and longer lookup paths.
  • Analytics adds write amplification on every redirect but provides business value.
