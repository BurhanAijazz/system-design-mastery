Pastebin Design

Pastebin is similar to a URL shortener but with content storage. The key difference is variable-size content up to 10 MB, which changes how we think about storage, caching, and delivery.

Define: Requirements

  • Functional: Create pastes (text up to 10 MB), unique short URL per paste, syntax highlighting, access control (public/unlisted/private), optional expiration
  • Non-Functional: High availability for reads, low latency (<100ms), durable storage, scale to millions of pastes

Key Differences from URL Shortener

# Comparison:
#                  | URL Shortener     | Pastebin
# -----------------+-------------------+--------------------
# Data size        | ~500 bytes        | 1KB - 10MB
# Content storage  | URL in DB         | Content in S3
# Read pattern     | Redirect (fast)   | Render page (slower)
# Write pattern    | Tiny metadata     | Large content upload
# Caching          | Easy (small)      | Harder (large)
# Access control   | None (public)     | Public/unlisted/private

# KEY INSIGHT: store METADATA in the database,
# store CONTENT in object storage (S3).

Estimate: Scale

# Assumptions: 1M new pastes/day, Read:Write = 5:1
# Average paste size: 10KB (median), up to 10MB (max)

# QPS:
#   Write: 1M / 86400 ~ 12 QPS (peak ~36)
#   Read:  5M / 86400 ~ 58 QPS (peak ~174)
# Much lower than URL shortener!

# Storage:
#   Daily metadata: 1M x 200B = 200 MB/day
#   Daily content:  1M x 10KB = 10 GB/day (on S3)
#   5-year content: 18.25 TB

# Cache: Top 20% of reads: 1M x 10KB = 10 GB
# CDN handles large pastes efficiently

Architecture

  • API Gateway: Routes requests, handles auth for private pastes
  • Metadata DB (PostgreSQL): paste_id, title, language, timestamps, access_level, user_id
  • Content Store (S3): Actual paste content stored as objects keyed by paste_id
  • CDN: Caches popular pastes at edge locations for fast delivery
  • Cache (Redis): Metadata cache plus hot paste content

Write Flow

  • Client uploads paste content to the API
  • Service generates a unique paste_id (same Base62 approach as URL shortener)
  • Content written to S3, metadata written to PostgreSQL
  • Return short URL to client

Read Flow

  • Client requests paste by short URL
  • Check CDN cache first, then Redis, then DB + S3
  • Render page with syntax highlighting based on language metadata
  • Access control check: public serves immediately, private requires auth

Design Considerations

  • Expiration: TTL-based cleanup using a background job scanning for expired pastes
  • Burn-after-read: Delete paste after first successful read (requires atomic check-and-delete)
  • Rate limiting: Prevent abuse by limiting paste creation per IP/user
  • Content moderation: Scan content for malware or policy violations asynchronously
