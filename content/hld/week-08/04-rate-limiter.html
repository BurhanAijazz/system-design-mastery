Distributed Rate Limiter Design

A rate limiter controls the number of requests a client can send within a time window. The challenge is making it work correctly across multiple API server nodes while adding less than 1ms of latency to every request.

Define: Requirements

  • Functional: Limit requests per user/IP/API-key within a time window; return 429 when exceeded; support tiered limits (free: 100/min, pro: 1000/min); provide rate limit headers
  • Non-Functional: Low latency (<1ms), high availability (fail-open if limiter fails), accurate counting in distributed mode, handle 100K+ checks/sec
  • Placement: API Gateway (most common), per-service, or client-side (advisory only)

Estimate

# 10M unique users, ~100 requests/min each
# Total: ~12,000 QPS average, ~36,000 peak
# Each rate check: ~200 bytes (request + response)
# Redis: 10M users x 100 bytes = 1 GB (single instance)
# Bandwidth: 36K QPS x 200B = 7.2 MB/s (trivial)

Rate Limiting Algorithms

Five main algorithms, each with different trade-offs:

1. Token Bucket

class TokenBucket:
    """Tokens refill at fixed rate, each request consumes one."""
    def __init__(self, capacity: int, refill_rate: float):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate  # tokens per second
        self.last_refill = time.time()

    def allow(self) -> bool:
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity,
                          self.tokens + elapsed * self.refill_rate)
        self.last_refill = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

Pros: Allows bursts up to capacity, smooth rate over time. Cons: Two parameters to tune (capacity and refill rate).

2. Fixed Window Counter

Count requests in fixed time windows (e.g., per minute). Simple but has a boundary problem: a user can send 2x the limit at the window boundary.

3. Sliding Window Counter (Recommended)

# Hybrid approach - best overall:
# Weighted count = (prev_window_count * overlap%) + curr_window_count
# Example: limit 100/min, at 0:01:15 (25% into current window)
#   prev window had 80 requests, current has 30
#   weighted = 80 * 0.75 + 30 = 90 -> ALLOW (under 100)

Pros: Smooth, no boundary spikes, memory efficient. Cons: Approximate (but very close to exact).

Distributed Architecture

  • Centralized Redis: All API servers check a shared Redis instance. Simple, consistent, but adds network hop.
  • Local + Sync: Each node tracks locally, periodically syncs to central store. Faster but may over-count temporarily.
  • Fail-open policy: If Redis is unreachable, allow the request rather than blocking all traffic.

Key Design Decisions

  • Rate limit headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset help clients self-throttle
  • Tiered limits: Store per-user tier in cache, look up rate limit configuration by tier
  • Race conditions: Use Redis INCR (atomic) or Lua scripts for check-and-increment atomicity
  • Monitoring: Track how many requests are being rate-limited to detect abuse or misconfigured limits
