Week 8 Exercises: Core Internet Services

These exercises reinforce the URL shortener, Pastebin, and rate limiter designs covered this week. Work through each one using the DERSC framework.

Exercise 1: URL Shortener with Analytics Dashboard

Design an extension to the URL shortener that provides real-time click analytics. Consider the following requirements:

  • Track every click: timestamp, referrer, country (via IP geolocation), device type
  • Provide dashboards showing clicks over time, top referrers, geographic distribution
  • Analytics queries should not slow down the redirect path

Tasks:

  • Estimate the additional storage for analytics events at 1B redirects/day
  • Design the data pipeline: how do click events flow from redirect to dashboard?
  • Choose between synchronous logging (inline with redirect) vs asynchronous (message queue). What are the trade-offs?
  • Sketch the analytics read path: pre-aggregated counters vs query-time aggregation

Exercise 2: Pastebin with Collaboration

Extend the Pastebin design to support real-time collaborative editing, similar to Google Docs but for code snippets.

  • Multiple users can edit the same paste simultaneously
  • Changes appear in real-time for all connected editors
  • Conflict resolution when two users edit the same line

Tasks:

  • Compare two approaches: Operational Transformation (OT) vs CRDTs. Which is simpler for a paste-sized document?
  • Design the WebSocket connection management for real-time sync
  • How does the system handle a user going offline and reconnecting with stale state?
  • Estimate the message rate: 10 active editors, each typing at 5 characters/second

Exercise 3: Multi-Algorithm Rate Limiter Comparison

Implement and compare all five rate limiting algorithms under the same traffic pattern.

# Simulate this traffic pattern:
# - 50 users, each allowed 100 requests/minute
# - Normal users: steady 1-2 requests/sec
# - Bursty users: 50 requests in 2 seconds, then idle
# - Abusive users: constant 10 requests/sec

# For each algorithm, measure:
# 1. How many legitimate requests were wrongly rejected?
# 2. How many abusive requests were wrongly allowed?
# 3. Memory usage per user
# 4. Redis operations per rate check

Tasks:

  • Implement Token Bucket, Fixed Window, and Sliding Window Counter
  • Run the same traffic simulation through each algorithm
  • Create a comparison table of accuracy, memory, and latency
  • Which algorithm would you recommend for an API gateway handling mixed traffic? Justify your choice with data.
