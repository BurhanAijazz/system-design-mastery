API Protocols: REST vs GraphQL vs gRPC

Choosing the right API protocol is a fundamental architecture decision. Each protocol has different strengths and trade-offs around performance, flexibility, developer experience, and tooling. This lesson compares the three most common protocols used in modern distributed systems.

REST (Representational State Transfer)

REST is the most widely used API style. It uses HTTP methods to operate on resources identified by URLs. It is simple, well-understood, and has excellent tooling support. 

# REST API design for a bookstore

# GET /api/books              - List all books
# GET /api/books/42           - Get book with ID 42
# POST /api/books             - Create a new book
# PUT /api/books/42           - Update book 42 (full replace)
# PATCH /api/books/42         - Partial update of book 42
# DELETE /api/books/42        - Delete book 42
# GET /api/books/42/reviews   - Get reviews for book 42

# Example REST endpoint implementation (Flask-like pseudocode)

from dataclasses import dataclass
from typing import Optional


@dataclass
class Book:
    id: int
    title: str
    author: str
    isbn: str
    price: float
    pages: int
    publisher: str
    year: int
    cover_url: str
    description: str


class BookAPI:
    """REST API for books."""

    def get_book(self, book_id: int) -> dict:
        """GET /api/books/{id}
        Returns ALL fields of the book.
        Client cannot choose which fields to receive."""
        book = self._repo.find(book_id)
        return {
            "id": book.id,
            "title": book.title,
            "author": book.author,
            "isbn": book.isbn,
            "price": book.price,
            "pages": book.pages,
            "publisher": book.publisher,
            "year": book.year,
            "cover_url": book.cover_url,
            "description": book.description,
        }

    def get_book_with_reviews(self, book_id: int) -> dict:
        """GET /api/books/{id}?include=reviews
        Problem: over-fetching (get all book fields even if you only need title)
        Problem: under-fetching (need 2 requests for book + reviews)"""
        book = self.get_book(book_id)
        reviews = self._review_repo.find_by_book(book_id)
        book["reviews"] = [r.__dict__ for r in reviews]
        return book

GraphQL

GraphQL lets clients specify exactly what data they need. Instead of multiple endpoints, there is a single endpoint with a query language. The client controls the shape of the response. 

# GraphQL schema for the same bookstore
#
# type Book {
#   id: ID!
#   title: String!
#   author: Author!
#   isbn: String!
#   price: Float!
#   pages: Int
#   publisher: String
#   year: Int
#   coverUrl: String
#   description: String
#   reviews: [Review!]!
# }
#
# type Author {
#   id: ID!
#   name: String!
#   books: [Book!]!
# }
#
# type Review {
#   id: ID!
#   rating: Int!
#   comment: String
#   reviewer: String!
# }
#
# type Query {
#   book(id: ID!): Book
#   books(limit: Int, offset: Int): [Book!]!
#   author(id: ID!): Author
# }

# Client query - get exactly what you need:
GRAPHQL_QUERY_MOBILE = """
query {
  book(id: 42) {
    title
    author { name }
    price
    coverUrl
  }
}
"""
# Response contains ONLY requested fields - no over-fetching

# Client query - get book with reviews in ONE request:
GRAPHQL_QUERY_DETAIL = """
query {
  book(id: 42) {
    title
    author { name }
    price
    description
    reviews {
      rating
      comment
      reviewer
    }
  }
}
"""
# No under-fetching - one query gets everything needed


# Python GraphQL resolver (using graphene-like pseudo-code)

class BookResolver:
    def resolve_book(self, info, id: int) -> dict:
        book = self._repo.find(id)
        return book

    def resolve_reviews(self, book, info) -> list:
        # Only called if client requests reviews field
        return self._review_repo.find_by_book(book.id)

    def resolve_author(self, book, info) -> dict:
        # Only called if client requests author field
        return self._author_repo.find(book.author_id)

gRPC (Google Remote Procedure Call)

gRPC uses Protocol Buffers (protobuf) for serialization and HTTP/2 for transport. It is significantly faster than REST and GraphQL for service-to-service communication due to binary encoding and multiplexed connections.

Head-to-Head Comparison

# Feature comparison:
#
# Feature           | REST            | GraphQL         | gRPC
# ------------------|-----------------|-----------------|------------------
# Protocol          | HTTP/1.1        | HTTP/1.1        | HTTP/2
# Data format       | JSON (text)     | JSON (text)     | Protobuf (binary)
# Schema            | OpenAPI (opt)   | SDL (required)  | .proto (required)
# Caching           | HTTP caching    | Complex         | No HTTP caching
# Over-fetching     | Common          | Solved          | N/A (typed)
# Under-fetching    | Common          | Solved          | N/A (typed)
# Streaming         | SSE/WebSocket   | Subscriptions   | Native (4 types)
# Browser support   | Excellent       | Good            | Limited (grpc-web)
# Learning curve    | Low             | Medium          | High
# Tooling           | Excellent       | Good            | Good
# Performance       | Good            | Good            | Excellent
# Best for          | Public APIs     | Flexible UIs    | Microservices

When to Use Each Protocol

# Decision guide:

use_rest_when = [
    "Building a public-facing API consumed by third-party developers",
    "The API is resource-oriented (CRUD operations)",
    "HTTP caching (CDN, browser) is important",
    "You need maximum compatibility (any language, any client)",
    "Team is new to API design (lowest learning curve)",
]

use_graphql_when = [
    "Multiple client types need different data shapes (mobile vs web vs TV)",
    "The frontend team wants control over data fetching",
    "You have deeply nested, interconnected data (graph-like)",
    "Over-fetching or under-fetching is a measurable performance problem",
    "Rapid frontend iteration is more important than backend simplicity",
]

use_grpc_when = [
    "Service-to-service communication in a microservices architecture",
    "Low latency is critical (p99 < 10ms)",
    "You need streaming (real-time data, large file transfers)",
    "Strong typing and code generation are valued",
    "Bandwidth efficiency matters (mobile, IoT, high-throughput)",
]

# Common hybrid approach (used by many large companies):
# External API:     REST or GraphQL (for clients)
# Internal services: gRPC (for service-to-service)
# Real-time:        WebSockets or gRPC streaming

Key Takeaways

  • REST is the safe default for public APIs. Widely understood, excellent tooling, HTTP caching.
  • GraphQL solves over/under-fetching but adds query parsing complexity and makes caching harder.
  • gRPC is the best choice for internal service-to-service communication. Binary encoding + HTTP/2 = low latency.
  • Many production systems use a hybrid: GraphQL or REST for client-facing APIs, gRPC internally.
  • In system design interviews, justify your protocol choice based on the specific requirements.
