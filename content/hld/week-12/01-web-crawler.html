Google Search / Web Crawler Design

A web crawler discovers and indexes the internet by following links. The two core constraints are priority (crawl important pages first) and politeness (do not overload any single website). At Google scale, this means crawling billions of pages per day.

Define: Requirements

  • Functional: Crawl from seed URLs, extract and follow links, store content for indexing, respect robots.txt, handle deduplication (same page via different URLs), re-crawl for freshness
  • Non-Functional: Crawl billions of pages, politeness (rate limit per domain), horizontally scalable, fault-tolerant (no lost progress), priority-based crawling

Estimate

# Target: 1B pages/day
# 1B / 86400 = ~11,600 pages/sec
# Avg page size: 100 KB -> 1.16 GB/sec = 100 TB/day
# Links per page: ~50 -> 50B links discovered/day (mostly dupes)

Architecture

# Crawler pipeline:
# Seed URLs -> URL Frontier -> Fetcher (HTTP GET)
#                  ^                    |
#                  |              Parser (extract links + content)
#             URL Dedup                 |
#          (Bloom Filter)         Content Store (S3)
#               ^                       |
#               |                 Link Extractor
#               +--- new URLs ----------+

# URL Frontier has two layers:
#   Front queues (PRIORITY): important pages first
#   Back queues (POLITENESS): one queue per domain

URL Frontier

The URL Frontier is the brain of the crawler with two key responsibilities:

  • Priority: Crawl important pages first based on PageRank score, freshness needs, and link depth (prefer shallow pages)
  • Politeness: Rate limit per domain (max 1 request/sec per domain), separate queue per domain, round-robin across domain queues

URL Deduplication

# Same page reachable via different URLs:
# https://example.com/page
# https://example.com/page?ref=twitter
# https://example.com/Page (case difference)
# http://example.com/page (http vs https)

# Approach:
# 1. URL normalization (lowercase, remove tracking params, etc.)
# 2. Bloom filter for fast "probably seen" check (~10 bits per URL)
#    50B URLs * 10 bits = 62.5 GB (fits in memory of a cluster)
# 3. If Bloom filter says "maybe" -> exact check against URL DB

Content Deduplication

  • Many pages have the same content but different URLs (mirrors, syndication)
  • SimHash or MinHash to compute content fingerprint
  • Compare fingerprints to detect near-duplicate content
  • Skip re-indexing if content has not changed since last crawl

Robots.txt Compliance

  • Fetch and cache robots.txt for each domain before crawling
  • Respect Disallow rules for specific paths
  • Honor Crawl-delay directive when specified
  • Cache robots.txt with a TTL (re-fetch daily)

Fault Tolerance

  • URL Frontier backed by persistent queue (Kafka) so no URLs are lost on crash
  • Checkpoint crawl progress periodically
  • Idempotent fetching: re-crawling a URL is safe (just updates stored content)
  • Dead letter queue for permanently failing URLs (404, timeout after retries)

Key Design Decisions

  • Bloom filter for dedup: Trades small false positive rate for massive memory savings
  • Per-domain queues: Essential for politeness without sacrificing overall throughput
  • Prioritized re-crawling: News sites re-crawled hourly; static pages re-crawled monthly
  • Horizontal scaling: Partition domains across crawler nodes using consistent hashing
