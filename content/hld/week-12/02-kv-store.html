Distributed Key-Value Store Design

Designing a distributed KV store (like DynamoDB or Cassandra) is the "meta" system design question -- you are designing the infrastructure that other systems are built on. Key challenges include consistent hashing for partitioning, tunable consistency, replication, and conflict resolution.

Define: Requirements

  • Functional: put(key, value), get(key), delete(key), automatic data partitioning, tunable consistency (strong to eventual)
  • Non-Functional: High availability (always writable), low latency (single-digit ms), scalable to petabytes, durable (never lose committed data), partition tolerant

Estimate

# 100M DAU, 10:1 read:write ratio
# 500K writes/sec peak, 5M reads/sec peak
# Average key: 64 bytes, average value: 1 KB
# 100 TB total data, growing 10 TB/year

# Storage per Node:
#   Target 2 TB per node (SSD)
#   100 TB / 2 TB = 50 data nodes minimum
#   With replication factor 3: 150 nodes -> ~200 total

# Latency Budget:
#   Client -> Coordinator: 1 ms
#   Coordinator -> Replica: 1 ms
#   Disk write (SSD): 0.1 ms
#   Total write: ~3-5 ms (quorum of 3, wait for 2)

# Memory per Node:
#   Bloom filter: 10 bits per key -> 1 GB for 1B keys
#   Key index in memory: 64B * 30M keys/node = 2 GB

Consistent Hashing

# Why consistent hashing?
# Regular hash (key % N nodes) requires remapping ALL keys
# when adding/removing nodes.
#
# Consistent hashing: arrange nodes on a ring (0 to 2^128)
# Each key hashes to a point on the ring -> assigned to the
# next node clockwise.
#
# Adding a node: only ~1/N keys need to move (not all keys)
# Virtual nodes: each physical node maps to 100+ points on
# the ring for even distribution

import hashlib

class ConsistentHashRing:
    def __init__(self, nodes, vnodes=150):
        self.ring = {}
        self.sorted_keys = []
        for node in nodes:
            for i in range(vnodes):
                key = f"{node}:vn{i}"
                h = int(hashlib.md5(key.encode()).hexdigest(), 16)
                self.ring[h] = node
                self.sorted_keys.append(h)
        self.sorted_keys.sort()

Replication and Quorum

  • Replication factor (N): Each key stored on N nodes (typically 3)
  • Write quorum (W): Number of replicas that must acknowledge a write
  • Read quorum (R): Number of replicas that must respond to a read
  • Strong consistency: W + R > N (e.g., W=2, R=2, N=3)
  • Eventual consistency: W=1, R=1 (fast but may read stale data)

Conflict Resolution

  • Last-write-wins (LWW): Use timestamps; latest write wins. Simple but can lose data.
  • Vector clocks: Track causal ordering across replicas. Detect true conflicts (concurrent writes).
  • Application-level resolution: Return all conflicting versions to the client; let the application merge.

Storage Engine

  • LSM Tree (Log-Structured Merge Tree): Optimized for writes. Writes go to memtable (in-memory), periodically flushed to sorted SSTables on disk. Compaction merges SSTables.
  • Bloom filters: Per-SSTable Bloom filter to quickly determine if a key might be in that SSTable, avoiding unnecessary disk reads.

Key Design Decisions

  • CAP theorem: This system chooses AP (availability + partition tolerance) by default, with tunable consistency via quorum settings
  • Anti-entropy: Merkle trees detect and repair inconsistencies between replicas in the background
  • Hinted handoff: If a replica is down during a write, a neighbor temporarily stores the data and forwards it when the replica recovers
  • Gossip protocol: Nodes share cluster membership and health information via peer-to-peer gossip
