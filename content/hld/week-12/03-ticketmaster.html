Ticketmaster / Event Booking System Design

Ticketmaster is the hardest booking problem in tech: millions of users simultaneously trying to book a limited number of tickets for the same event. The core challenge is extreme write contention on a single resource -- a hot partition problem on steroids.

Define: Requirements

  • Functional: Browse events, view available seats, select and hold seats temporarily (5 min), complete purchase within hold period, release tickets if payment fails/times out, prevent double-booking
  • Non-Functional: Handle 10M+ concurrent users for hot events, fairness (first-come-first-served or lottery), seat hold <500ms, strong consistency for inventory, graceful degradation under load

The Core Challenge

# Taylor Swift concert: 70,000 seats, 14 MILLION users
# That's 200 users per seat, all clicking at the same second
# This is a HOT PARTITION problem on steroids.

# Normal DB: ~5K writes/sec per partition -> instant bottleneck
# All 33K QPS target the SAME event

Estimate

# Mega event (first 10 minutes):
#   Page views: 2M concurrent x 10 refreshes = 20M requests
#   QPS peak: 20M / 600s = ~33K QPS on ONE event
#   Seat hold attempts: 2M users -> 500K/min -> 8.3K/sec
#   Payment: 70K successful in 10-30 min

# Storage is NOT the problem:
#   Event metadata: 500 events x 10 KB = 5 MB
#   Seat inventory: 70K seats x 200 bytes = 14 MB
#   Booking records: 70K x 1 KB = 70 MB

# The real challenge is CONCURRENCY, not storage.

Architecture: Virtual Waiting Room

  • Queue-based admission: Instead of letting all 14M users hit the booking system, place them in a virtual queue
  • Controlled admission rate: Release users from the queue at a rate the booking system can handle (e.g., 5K/sec)
  • Position tracking: Each user sees their queue position and estimated wait time
  • Fairness: Queue position assigned randomly or by arrival time

Seat Inventory Management

# Approach: Pre-partition the seat inventory
#
# Stadium has 5 sections of 14K seats each
# Each section managed by a separate service instance
# User hash determines which section they can browse first
#
# Within each section:
# 1. Seat HOLD: 5-minute temporary reservation
#    Uses Redis with TTL for automatic expiry
# 2. Seat PURCHASE: convert hold to confirmed booking
#    Uses DB transaction with optimistic locking
# 3. Seat RELEASE: TTL expires or explicit cancel
#    Seat returns to available pool

# This distributes the hot partition:
# Instead of 8.3K writes/sec on one partition,
# 5 partitions handle 1.66K writes/sec each

Handling Extreme Concurrency

  • Read path (seat map): Cache aggressively. Seat map updates every few seconds, not real-time. "Approximately N seats available" is acceptable.
  • Write path (seat hold): Optimistic locking with retries. If two users try the same seat, one succeeds, one retries on a different seat.
  • Payment timeout: 5-minute hold with automatic release. If payment takes too long, seat returns to pool.
  • Oversell prevention: The seat hold is the single source of truth. Only one hold can exist per seat at any time.

Lottery vs First-Come-First-Served

  • FCFS: Advantages early arrivals, penalizes slow connections. Creates server stampede.
  • Lottery: Registration window (e.g., 48 hours), random selection, winners get purchase window. Fairer, reduces peak load.
  • Hybrid: Presale for fan club members (FCFS within smaller group), general sale with lottery.

Key Design Decisions

  • Virtual queue is essential: Without it, the booking system would collapse under load
  • Temporary holds with TTL: Prevents seat hoarding; automatic cleanup without manual intervention
  • Approximate availability: Showing real-time exact seat counts is impossible under contention; approximate is sufficient
  • Graceful degradation: If system is overloaded, show "high demand" page rather than error pages
