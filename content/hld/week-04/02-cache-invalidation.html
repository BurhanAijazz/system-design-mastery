Cache Invalidation and Consistency

"There are only two hard things in Computer Science: cache invalidation and naming things." -- Phil Karlton. This lesson covers every cache failure mode and how to fix each one.

TTL (Time-To-Live): The Simplest Invalidation

Set an expiration time on each cache entry. After the TTL, the key is automatically deleted.

from dataclasses import dataclass
from typing import Dict, Optional

@dataclass
class TTLCacheEntry:
    value: str
    expires_at: float

class TTLCache:
    """Cache with TTL-based expiration."""
    def __init__(self, ttl_seconds: float):
        self.ttl = ttl_seconds
        self.store: Dict[str, TTLCacheEntry] = {}
        self.hits = 0
        self.misses = 0

    def get(self, key: str, current_time: float) -> Optional[str]:
        entry = self.store.get(key)
        if entry and current_time < entry.expires_at:
            self.hits += 1
            return entry.value
        self.misses += 1
        return None

    def put(self, key: str, value: str, current_time: float):
        self.store[key] = TTLCacheEntry(value, current_time + self.ttl)

The TTL Trade-off

  • Short TTL (30s): fresher data, more DB hits, lower hit rate
  • Long TTL (24h): more stale data, fewer DB hits, higher hit rate
  • Choose based on: how often data changes, how bad stale data is, how expensive the DB query is

Event-Based Invalidation

When data changes in the database, actively invalidate or update the cache. More complex but ensures freshness.

  • Delete on write: When updating the DB, delete the cache key. Next read repopulates
  • Update on write: When updating the DB, update the cache too (write-through)
  • Pub/Sub invalidation: DB publishes change events, cache subscribers invalidate affected keys

Common Cache Failure Modes

Cache Stampede (Thundering Herd)

A popular cache key expires. Thousands of concurrent requests all miss the cache and hit the database simultaneously.

  • Fix 1: Locking -- Only one request fetches from DB; others wait for the cache to be populated
  • Fix 2: Stale-while-revalidate -- Serve stale data while one request refreshes in the background
  • Fix 3: Jittered TTL -- Add random offset to TTLs so keys don't all expire at once

Cache Penetration

Requests for keys that don't exist in the database bypass the cache every time.

  • Fix: Cache negative results -- Store "not found" with a short TTL
  • Fix: Bloom filter -- Check a bloom filter before querying the DB

Cache Avalanche

Many cache keys expire at the same time, causing a sudden spike in database load.

  • Fix: Jittered TTLs, multi-layer caching, circuit breaker on the DB

Hot Key Problem

A single key receives disproportionate traffic, overloading one cache node.

  • Fix: Replicate the hot key across multiple cache nodes, or use local in-process cache
