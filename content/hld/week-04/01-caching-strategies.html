Caching Strategies: Read and Write Patterns

Caching is the single most impactful optimization in system design. It trades memory for speed and reduced database load. There are five patterns, each with different trade-offs for consistency, performance, and complexity.

Simulated Database and Cache

To understand caching patterns, consider two components: a slow database (~50ms latency) and a fast cache (~1ms latency).

from collections import OrderedDict
from typing import Dict, Optional, Any

class SimulatedDB:
    """Database with artificial latency (~50ms)."""
    def __init__(self, latency_ms=50.0):
        self.store: Dict[str, Any] = {}
        self.read_count = 0
        self.write_count = 0

    def get(self, key: str) -> Optional[Any]:
        self.read_count += 1
        return self.store.get(key)

    def put(self, key: str, value: Any) -> None:
        self.write_count += 1
        self.store[key] = value

class SimulatedCache:
    """In-memory LRU cache (~1ms latency)."""
    def __init__(self, max_size=100):
        self.store: OrderedDict = OrderedDict()
        self.hits = 0
        self.misses = 0

    def get(self, key: str) -> Optional[Any]:
        if key in self.store:
            self.hits += 1
            self.store.move_to_end(key)  # LRU: move to back
            return self.store[key]
        self.misses += 1
        return None

    def put(self, key: str, value: Any) -> None:
        self.store[key] = value
        if len(self.store) > 100:
            self.store.popitem(last=False)  # Evict oldest

Pattern 1: Cache-Aside (Lazy Loading)

The application checks the cache first. On a miss, it reads from the database and populates the cache. This is the most common pattern.

  • Pro: Only caches data that is actually requested
  • Con: First request is always a cache miss (cold start)
  • Con: Data can become stale if updated in the database

Pattern 2: Write-Through

Every write goes to both the cache and the database. Ensures the cache is always consistent with the database.

  • Pro: Cache is never stale
  • Con: Higher write latency (two writes per operation)
  • Con: Caches data that may never be read

Pattern 3: Write-Behind (Write-Back)

Writes go to the cache immediately, then asynchronously flushed to the database in batches.

  • Pro: Very fast writes (cache speed)
  • Pro: Batching reduces database load
  • Con: Risk of data loss if cache crashes before flush

Pattern 4: Read-Through

The cache itself handles loading from the database on a miss. The application only talks to the cache.

Pattern 5: Refresh-Ahead

The cache proactively refreshes entries before they expire, based on predicted access patterns.

Choosing the Right Pattern

  • Cache-Aside: general purpose, read-heavy workloads
  • Write-Through: when you need strong cache consistency
  • Write-Behind: write-heavy workloads where brief inconsistency is OK
  • Read-Through: simplifies application code, common with CDNs
  • Refresh-Ahead: latency-sensitive reads with predictable access patterns
