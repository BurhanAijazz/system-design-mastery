CDN and Edge Caching

A request passes through multiple cache layers before hitting your database. Each layer catches traffic, reducing load on the next. Understanding this hierarchy is essential for designing performant systems.

Multi-Layer Caching Hierarchy

  • Layer 1 -- Browser Cache: Cached in user's browser. Zero network latency. Best for static assets (CSS, JS, images). Controlled by Cache-Control headers. ~60% hit rate
  • Layer 2 -- CDN Edge: Cached at edge servers close to the user (~50ms instead of ~200ms). Hundreds of Points of Presence worldwide. Best for static assets, public API responses. ~25% of remaining traffic
  • Layer 3 -- API Gateway Cache: Per-route caching with custom TTLs. Best for expensive API responses
  • Layer 4 -- Application Cache (Redis): Caches DB query results, session data, computed values
  • Layer 5 -- Database Cache: Query cache, buffer pool, materialized views. Built into the DB engine
from dataclasses import dataclass
from typing import List

@dataclass
class CacheLayer:
    name: str
    hit_rate: float     # probability of cache hit
    latency_ms: float   # latency if this layer serves the response
    hits: int = 0
    misses: int = 0

def simulate_multi_layer(num_requests=100_000) -> List[CacheLayer]:
    layers = [
        CacheLayer("Browser Cache", hit_rate=0.55, latency_ms=0),
        CacheLayer("CDN Edge",      hit_rate=0.70, latency_ms=20),
        CacheLayer("API Gateway",   hit_rate=0.40, latency_ms=5),
        CacheLayer("Redis Cache",   hit_rate=0.90, latency_ms=2),
        CacheLayer("Database",      hit_rate=1.00, latency_ms=50),
    ]
    # Each request falls through layers until one hits
    # ~2% of original traffic actually reaches the database
    return layers

Cache-Control Headers

HTTP headers that control browser and CDN caching behavior:

  • Cache-Control: public, max-age=31536000 -- CDN and browser cache for 1 year (static assets)
  • Cache-Control: private, max-age=300 -- browser only, 5 minutes (user-specific data)
  • Cache-Control: no-cache -- always revalidate with origin (not "no caching")
  • Cache-Control: no-store -- truly do not cache (sensitive data)
  • ETag and If-None-Match -- conditional requests, return 304 Not Modified if unchanged

CDN Architecture

  • Pull CDN: CDN fetches from origin on first request, caches the response. Simple but first request is slow
  • Push CDN: You upload content to the CDN proactively. More control, better for large files
  • Major providers: Cloudflare, AWS CloudFront, Akamai, Fastly

CDN Invalidation Strategies

  • TTL-based: Set expiration, accept staleness window
  • Versioned URLs: style.v2.css -- new URL = new cache entry. The safest approach
  • Purge API: Explicitly invalidate specific URLs. Use sparingly -- propagation takes time

When to Use a CDN

  • Static assets (images, CSS, JS, fonts) -- always use a CDN
  • Public API responses that don't change frequently
  • Video and media streaming
  • Do NOT use for: user-specific data, real-time data, authenticated endpoints
