Database Types: KV, Document, Wide-Column, Graph

Each NoSQL database type optimizes for a specific access pattern. Choosing the wrong type means fighting against the database's design. In NoSQL, you design the schema around the queries you need to run, not around your entities.

Modeling a Social Network -- Four Ways

The same data (users, follows, posts, likes) modeled in four different database types reveals their strengths.

Relational Model (PostgreSQL)

class RelationalDB:
    """Simulate a relational database with tables and JOINs."""

    def __init__(self):
        self.users = {}       # id -> {name, email}
        self.follows = []     # (follower_id, following_id)
        self.posts = {}       # id -> {author_id, content}
        self.likes = []       # (user_id, post_id)

    def get_followers(self, user_id):
        """SELECT u.* FROM users u JOIN follows f
           ON u.id = f.follower_id WHERE f.following_id = ?"""
        follower_ids = [f[0] for f in self.follows if f[1] == user_id]
        return [self.users[fid] for fid in follower_ids]

    def get_feed(self, user_id, limit=10):
        """JOIN posts with follows to build a feed."""
        following = {f[1] for f in self.follows if f[0] == user_id}
        feed = [p for p in self.posts.values()
                if p["author_id"] in following]
        return feed[:limit]

Document Model (MongoDB)

Embed related data directly in the document. No JOINs needed for common reads, but updates to shared data require touching many documents.

# Document model: self-contained user document
user_doc = {
    "_id": "user:1",
    "name": "Alice",
    "email": "alice@example.com",
    "followers": ["user:2", "user:3"],
    "posts": [
        {"id": "post:1", "content": "Hello!", "likes": 42},
        {"id": "post:2", "content": "World!", "likes": 17},
    ]
}
# Reading a user profile = 1 document read (fast!)
# But updating a follower's name requires updating many documents

Graph Model (Neo4j)

Nodes and edges. Optimized for traversing relationships -- finding friends-of-friends, shortest paths, and network patterns.

  • Best for: social networks, recommendation engines, fraud detection
  • Query "friends of friends" is O(depth) not O(n) like SQL JOINs

Wide-Column Model (Cassandra)

Rows with flexible columns, optimized for massive write throughput and time-series data. Each row can have different columns.

  • Best for: IoT sensor data, activity logs, messaging at scale
  • Writes distributed across nodes with tunable consistency

Polyglot Persistence

Real-world systems use multiple database types together:

  • PostgreSQL for user accounts and payments (ACID needed)
  • Redis for sessions and caching (speed needed)
  • Elasticsearch for product search (full-text search)
  • Cassandra for activity feeds (high write volume)
  • Neo4j for recommendations (relationship traversal)

Choosing the Right Database

  • Model your data based on your queries, not your entities
  • Start with PostgreSQL if unsure -- it handles most patterns well
  • Add specialized databases only when you have a clear access pattern that demands it
