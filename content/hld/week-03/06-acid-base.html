ACID vs BASE: Transactions and Consistency Models

Transactions determine how your system behaves under concurrent access and failure. The question isn't "ACID or BASE?" -- it's which parts of your system need ACID (payments, inventory) and which can tolerate BASE (like counts, recommendations)?

ACID Transactions -- The Gold Standard

  • Atomicity: All or nothing. A transaction either fully completes or fully rolls back
  • Consistency: The database moves from one valid state to another. All constraints are satisfied
  • Isolation: Concurrent transactions don't interfere with each other
  • Durability: Once committed, data survives crashes and power failures
class SimpleTransactionDB:
    """Demonstrate ACID atomicity with a bank transfer."""

    def __init__(self):
        self.accounts = {"Alice": 1000, "Bob": 500}
        self.wal = []  # Write-Ahead Log for durability

    def transfer(self, from_acct, to_acct, amount):
        """Atomic transfer: both succeed or both fail."""
        # Begin transaction
        snapshot = dict(self.accounts)
        try:
            if self.accounts[from_acct] < amount:
                raise ValueError("Insufficient funds")
            self.accounts[from_acct] -= amount  # Debit
            self.accounts[to_acct] += amount    # Credit
            self.wal.append(f"COMMIT: {from_acct}->{to_acct} ${amount}")
        except Exception:
            self.accounts = snapshot  # ROLLBACK
            self.wal.append(f"ROLLBACK: {from_acct}->{to_acct} ${amount}")
            raise

Isolation Levels

Isolation levels trade consistency for performance. From strongest to weakest:

  • Serializable: Full isolation. Transactions execute as if serial. Slowest
  • Repeatable Read: No dirty or non-repeatable reads. Phantom reads possible
  • Read Committed: No dirty reads. Most common default (PostgreSQL)
  • Read Uncommitted: Can see uncommitted data. Fastest but risky

BASE Properties

BASE stands for Basically Available, Soft state, Eventually consistent. Used by NoSQL systems that prioritize availability over consistency.

  • Basically Available: System always responds, even with stale data
  • Soft state: Data may change over time without input (replication lag)
  • Eventually consistent: Given enough time, all replicas converge

When to Use ACID vs BASE

  • ACID: Bank transfers, inventory management, booking systems, order processing
  • BASE: Like counts, view counters, recommendations, analytics, activity feeds

Distributed Transactions (2PC)

Two-Phase Commit coordinates transactions across multiple databases. Phase 1: prepare (all vote yes/no). Phase 2: commit or abort based on votes.

# 2PC conceptual flow
# Phase 1: PREPARE
coordinator.send_all("PREPARE transaction_id=123")
votes = [participant.vote() for participant in participants]

# Phase 2: COMMIT or ABORT
if all(v == "YES" for v in votes):
    coordinator.send_all("COMMIT transaction_id=123")
else:
    coordinator.send_all("ABORT transaction_id=123")
# Problem: if coordinator crashes between phases, participants are stuck

Why Microservices Avoid 2PC

2PC is slow, fragile, and creates tight coupling. Modern microservices use the Saga pattern instead: a sequence of local transactions with compensating actions on failure.
