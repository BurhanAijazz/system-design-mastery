SQL vs NoSQL: When to Use Which

The database is the most critical building block in your system. The wrong choice can mean rewriting your entire data layer. The right choice depends on your access patterns, not on hype.

Most production systems use multiple databases (polyglot persistence). The real question is: which database for which part of your system?

Relational Databases (SQL) -- The Default Choice

Examples: PostgreSQL, MySQL, Oracle, SQL Server. Data is stored in tables with rows and columns, strict schemas, and relationships via foreign keys.

# Relational model: tables with relationships
users = {
    1: {"name": "Alice", "email": "alice@example.com"},
    2: {"name": "Bob",   "email": "bob@example.com"},
}
orders = {
    1: {"user_id": 1, "amount": 50.00},
    2: {"user_id": 1, "amount": 30.00},
    3: {"user_id": 2, "amount": 100.00},
}

# SQL JOIN equivalent
# SELECT u.name, o.amount FROM users u
#   JOIN orders o ON u.id = o.user_id
for oid, order in orders.items():
    user = users[order["user_id"]]
    print(f"{user['name']} -> ${order['amount']}")

Strengths and Weaknesses of SQL

  • ACID transactions -- atomicity, consistency, isolation, durability
  • Complex queries -- JOINs, aggregations, subqueries
  • Schema enforcement -- data integrity, can't insert garbage
  • Mature ecosystem -- decades of optimization and tooling
  • Weakness: Vertical scaling ceiling (single node for writes)
  • Weakness: Schema changes can be painful at scale
  • Weakness: Not great for hierarchical or graph-like data

Key-Value Stores -- The Fastest Option

Examples: Redis, DynamoDB, Memcached. Simple key-to-value mapping with O(1) lookups. Best for caching, sessions, and real-time counters.

# Key-Value store: simple get/set
cache = {}
cache["user:123"] = '{"name": "Alice", "email": "alice@example.com"}'
cache["session:abc"] = '{"user_id": 123, "expires": 1700000000}'

# O(1) lookup -- no scanning, no indexes needed
user_data = cache.get("user:123")

Document Stores

Examples: MongoDB, Couchbase. Store self-contained JSON documents. Ideal when entities are mostly read/written as a whole and don't need complex JOINs.

Wide-Column and Graph Databases

  • Wide-Column (Cassandra, HBase): optimized for massive write throughput and time-series data
  • Graph (Neo4j, Neptune): optimized for traversing relationships (social networks, fraud detection)

Decision Flowchart

  • Need ACID transactions and complex JOINs? Use SQL (PostgreSQL, MySQL)
  • Need sub-millisecond lookups by key? Use Key-Value (Redis, DynamoDB)
  • Data is self-contained documents with variable schema? Use Document (MongoDB)
  • Need massive write throughput across regions? Use Wide-Column (Cassandra)
  • Need to traverse deep relationships? Use Graph (Neo4j)
  • Unsure? Start with PostgreSQL -- it handles most workloads well

Scale Limits (Approximate)

  • Single PostgreSQL: ~10K-50K QPS, ~1TB comfortable
  • With read replicas: ~200K read QPS
  • Beyond that: need sharding or move writes to a specialized store
