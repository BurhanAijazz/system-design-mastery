Replication: Leader-Follower, Multi-Leader, Leaderless

Replication copies your data to multiple machines. It provides high availability (survive server failures), read scaling (distribute read load), and geographic proximity (serve users from nearby replicas).

Every replication strategy is a trade-off between consistency, availability, and latency.

Why Replicate?

  • High Availability: If the primary dies, a replica takes over
  • Read Scaling: 1 server handles 10K QPS; add 4 replicas for 50K QPS
  • Geographic Proximity: EU users read from EU replica (20ms vs 200ms)

Single-Leader (Master-Slave) Replication

The most common strategy. One leader accepts writes and replicates to N followers.

class SingleLeaderCluster:
    """Simulate single-leader replication."""

    def __init__(self, num_followers=3):
        self.leader_data = {}
        self.follower_data = [{} for _ in range(num_followers)]
        self.replication_lag = 0

    def write(self, key, value):
        """Writes always go to the leader."""
        self.leader_data[key] = value
        # Async replication to followers (with lag)
        for follower in self.follower_data:
            follower[key] = value  # simplified: instant here

    def read_from_leader(self, key):
        """Strong consistency: always read latest."""
        return self.leader_data.get(key)

    def read_from_follower(self, key, follower_idx=0):
        """Eventual consistency: may read stale data."""
        return self.follower_data[follower_idx].get(key)

Synchronous vs Asynchronous Replication

  • Synchronous: Leader waits for follower to confirm write. Strong consistency but slower writes
  • Asynchronous: Leader returns success immediately. Fast writes but stale reads possible
  • Semi-synchronous: Wait for 1 of N followers. Balance of durability and speed

Multi-Leader Replication

Multiple nodes accept writes, typically in multi-datacenter setups. Each datacenter has its own leader.

  • Pro: Writes succeed even if one datacenter is unreachable
  • Con: Write conflicts -- two leaders may modify the same data
  • Conflict resolution: Last-writer-wins (LWW), custom merge logic, or CRDTs

Leaderless (Dynamo-Style) Replication

Any node accepts reads and writes. Uses quorum-based consistency.

# Quorum formula: W + R > N
# N = total replicas, W = write quorum, R = read quorum
#
# Example: N=3, W=2, R=2
# Write succeeds when 2 of 3 nodes confirm
# Read queries 2 of 3 nodes, guaranteed to see latest write

N, W, R = 3, 2, 2
assert W + R > N  # This guarantees overlap -> consistency

Replication Lag Consequences

  • Read-after-write inconsistency: User writes data, immediately reads from stale replica
  • Monotonic read violation: User sees newer data, refreshes, sees older data from different replica
  • Solution: Read-your-own-writes guarantee -- route user to leader for recently written data
