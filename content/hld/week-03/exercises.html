Exercise 2: Sharding Strategy Design

You have an e-commerce platform with 500 million orders. The single database is at capacity. Design a sharding strategy.

  • Consider the most common queries: lookup order by order_id, list orders by user_id, aggregate sales by date range
  • What shard key would you choose? Why?
  • How would you handle the query "get all orders for user X" if the shard key is order_id?
  • What happens when you need to add more shards?
# Compare two shard key strategies:
import hashlib

def shard_by_order_id(order_id: str, num_shards: int) -> int:
    """Shard by order_id: even distribution, but user queries scatter."""
    return int(hashlib.md5(order_id.encode()).hexdigest(), 16) % num_shards

def shard_by_user_id(user_id: str, num_shards: int) -> int:
    """Shard by user_id: user queries hit one shard, but risk hot users."""
    return int(hashlib.md5(user_id.encode()).hexdigest(), 16) % num_shards

# What secondary indexes or routing tables might you need?

Exercise 3: Replication Failure Scenarios

You run a PostgreSQL cluster with 1 leader and 3 followers using asynchronous replication. Walk through each scenario:

  • Scenario A: A user updates their profile (write to leader), then immediately refreshes the page (read from a follower). What might they see? How do you fix this?
  • Scenario B: The leader crashes. Follower 2 has a 5-second replication lag. Follower 1 is up to date. Which follower should be promoted? What data is lost if you promote Follower 2?
  • Scenario C: You switch to synchronous replication for all 3 followers. What happens to write latency? What if one follower is in a different region with 100ms network latency?
# Think about these consistency guarantees:
# 1. Read-after-write: user sees their own writes
# 2. Monotonic reads: user never sees data go "backward"
# 3. Consistent prefix: causally related writes seen in order

# For each scenario, identify which guarantee is violated
# and propose a fix (e.g., sticky sessions, read-from-leader,
# semi-synchronous replication).
