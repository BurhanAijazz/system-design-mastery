Sharding Strategies

When a single database cannot handle your write load or data volume, you split the data across multiple database instances. This is sharding. Replication scales reads; sharding scales writes and storage.

When to Shard

Sharding is hard. Avoid it as long as possible by following this progression:

  • First: optimize queries and add indexes
  • Then: add read replicas for read scaling
  • Then: add a caching layer (Redis)
  • Finally: shard when you truly must (write bottleneck or data too big)

Sharding Strategies

Range-Based Sharding

Split by value ranges of the shard key. Simple but can create hot partitions.

def range_shard(user_id: int, shard_ranges: list) -> str:
    """Route to shard based on ID range."""
    # shard_ranges = [(0, 250000, "shard-0"), (250001, 500000, "shard-1"), ...]
    for low, high, shard_name in shard_ranges:
        if low <= user_id <= high:
            return shard_name
    return "shard-overflow"

Hash-Based Sharding

Hash the shard key and mod by number of shards. Even distribution but no range queries.

import hashlib

def hash_shard(key: str, num_shards: int) -> int:
    """Route to shard using hash of key."""
    h = int(hashlib.md5(key.encode()).hexdigest(), 16)
    return h % num_shards

# Even distribution across 4 shards
for user_id in range(1000):
    shard = hash_shard(f"user:{user_id}", num_shards=4)

Directory-Based Sharding

A lookup table maps each key to its shard. Most flexible but the directory is a single point of failure.

Geographic Sharding

Route by region for data locality. EU users go to EU shard, US users to US shard.

Shard Key Selection

The shard key is the most important decision in sharding. A bad key leads to hot partitions and cross-shard queries.

  • High cardinality: many unique values for even distribution
  • Even distribution: avoids hot partitions where one shard gets all traffic
  • Query alignment: most queries should hit a single shard
  • Bad keys: timestamp (all new writes go to one shard), country (uneven population)
  • Good keys: user_id, order_id (high cardinality, even distribution)

Cross-Shard Queries

JOINs across shards are expensive -- they require scatter-gather across all shards. Design your shard key so most queries hit a single shard.

The Hot Partition Problem

Even with hash-based sharding, a single hot key (celebrity user, viral post) can overwhelm one shard. Solutions include further splitting the hot key or adding a caching layer in front.

Resharding

Adding or removing shards with hash-based sharding reshuffles most keys. Use consistent hashing to minimize data movement when the number of shards changes.
