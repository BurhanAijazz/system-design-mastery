Project: Sharding Simulator

This project is an interactive simulator that demonstrates sharding strategies, data distribution, migration costs, and query routing. It builds intuition about the trade-offs between range, hash, and consistent-hash sharding.

Core Data Models

from dataclasses import dataclass, field
from typing import Dict, List
from abc import ABC, abstractmethod
import hashlib

@dataclass
class Record:
    """A synthetic data record to be sharded."""
    key: str
    value: str = ""
    timestamp: int = 0

@dataclass
class Shard:
    """Represents a database shard."""
    name: str
    records: int = 0

@dataclass
class ShardingResult:
    """Result of distributing data across shards."""
    strategy_name: str
    total_records: int
    distribution: Dict[str, int]
    std_dev: float = 0.0
    hottest_shard: str = ""
    hottest_pct: float = 0.0

@dataclass
class MigrationResult:
    """Result of adding/removing a shard."""
    records_moved: int
    total_records: int
    migration_pct: float

Sharding Strategy Interface

class ShardingStrategy(ABC):
    """Base class for sharding strategies."""

    @abstractmethod
    def assign(self, key: str) -> str:
        """Return the shard name for a given key."""
        pass

    @abstractmethod
    def add_shard(self, shard_name: str) -> None:
        pass

    @abstractmethod
    def remove_shard(self, shard_name: str) -> None:
        pass

Key Strategies to Implement

  • Range-Based: Assign keys by value ranges. Simple but creates hot partitions for sequential inserts
  • Hash-Based: hash(key) % num_shards. Even distribution but reshuffles all keys when shard count changes
  • Consistent Hashing: Place shards on a ring. Only ~1/N keys move when a shard is added or removed

What to Compare

  • Distribution evenness: Standard deviation of records per shard. Lower is better
  • Migration cost: Percentage of keys that move when adding a shard
  • Hot partition risk: Percentage of data on the busiest shard

Sample Experiment

import statistics
from collections import defaultdict

def measure_distribution(strategy, keys, shard_names):
    """Measure how evenly keys are distributed."""
    counts = defaultdict(int)
    for key in keys:
        shard = strategy.assign(key)
        counts[shard] += 1

    values = [counts[s] for s in shard_names]
    std = statistics.stdev(values) if len(values) > 1 else 0
    hottest = max(values)
    print(f"  Std dev: {std:.1f}")
    print(f"  Hottest shard: {hottest}/{len(keys)} "
          f"({hottest/len(keys)*100:.1f}%)")

def measure_migration(strategy, keys, old_shards, new_shards):
    """Measure keys moved when adding a shard."""
    before = {k: strategy.assign(k) for k in keys}
    strategy.add_shard(new_shards[-1])
    after = {k: strategy.assign(k) for k in keys}
    moved = sum(1 for k in keys if before[k] != after[k])
    print(f"  Keys moved: {moved}/{len(keys)} ({moved/len(keys)*100:.1f}%)")

Expected Results

  • Hash-based: even distribution (~25% per shard with 4 shards) but ~75% migration when adding a shard
  • Consistent hashing: slightly less even but only ~20-25% migration when adding a shard
  • Range-based: uneven if keys are skewed, but range queries are efficient
