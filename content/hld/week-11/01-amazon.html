Amazon / E-Commerce Platform Design

An e-commerce platform like Amazon is a microservices-by-domain architecture covering product catalog, search, cart, checkout, inventory management, payment idempotency, and flash sale handling. The key challenge is strong consistency at checkout (no overselling) while maintaining high availability everywhere else.

Define: Requirements

  • Functional: Product catalog (browse, details), search with filters and autocomplete, shopping cart, checkout and payment, order management (track/cancel/return), reviews and ratings, seller management, recommendations
  • Non-Functional: Search <200ms, cart ops <100ms, checkout <2 seconds, handle flash sales (100x traffic), payment idempotency (never double-charge), inventory strong consistency, 99.99% availability for cart and checkout

Estimate

# 300M active customers, 2.5M sellers, 1.5B products
# 50M DAU, 5M orders/day (20M on Prime Day)

# READ:
#   Product views: 50M x 20 pages = 1B/day = 11,600 QPS
#   Search: 50M x 5 = 250M/day = 2,900 QPS

# WRITE:
#   Cart: 50M x 10 actions = 500M/day = 5,800 QPS
#   Orders: 5M/day = 58 QPS (but 5-10 ops each)

# FLASH SALE (Prime Day):
#   Product views: 100K QPS (10x normal)
#   Cart: 50K QPS
#   Orders: 500 QPS -> all for SAME products = hot partition

# STORAGE:
#   Product catalog: 1.5B x 10 KB = 15 TB
#   Product images: 1.5B x 5 x 500 KB = 3.75 PB (S3 + CDN)

Architecture: Microservices by Domain

  • Product Service: Catalog CRUD, product details. PostgreSQL sharded by category.
  • Search Service: Elasticsearch with autocomplete, faceted search, filters.
  • Cart Service: Redis for active carts (fast), DynamoDB for persistence. Cart survives session expiry.
  • Inventory Service: Strong consistency required. Uses pessimistic locking or compare-and-swap at checkout.
  • Order Service: Orchestrates the checkout saga across inventory, payment, and fulfillment.
  • Payment Service: Idempotent payment processing with idempotency keys.
  • Recommendation Service: Collaborative filtering, "customers also bought" computed offline.

Checkout Saga Pattern

# The checkout is a distributed transaction (saga):
# Step 1: Reserve inventory (lock items)
# Step 2: Process payment (charge card)
# Step 3: Confirm order (update status)
# Step 4: Notify fulfillment (ship it)

# If Step 2 fails (payment declined):
#   Compensate Step 1: release inventory
# If Step 3 fails:
#   Compensate Step 2: refund payment
#   Compensate Step 1: release inventory

# Each step is idempotent - safe to retry on failure

Flash Sale Architecture

  • Problem: Millions of users trying to buy the same product simultaneously creates a hot partition
  • Pre-warm caches: Product page and inventory counts cached in Redis before the sale starts
  • Queue-based ordering: Place users in a virtual queue; process orders sequentially to prevent overselling
  • Inventory pre-partitioning: Split 10,000 units across 100 partitions (100 units each) to distribute write contention

Key Design Decisions

  • Eventual consistency for catalog: Product details can be stale by a few seconds without harm
  • Strong consistency for inventory: Overselling is unacceptable; use distributed locks or serialized queues
  • Payment idempotency: Every payment request includes an idempotency key; retries are safe
  • Cart in Redis: Sub-millisecond cart operations; persist to DynamoDB asynchronously for durability
