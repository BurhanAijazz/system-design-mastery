Week 11 Exercises: Commerce and Platforms

These exercises explore e-commerce flash sales, geospatial matching, and booking systems covered this week.

Exercise 1: Flash Sale Inventory System

Design the inventory management system for an Amazon Prime Day-style flash sale where 1 million users try to buy 10,000 units of a single product within 60 seconds.

  • No overselling: exactly 10,000 orders, not 10,001
  • Fairness: first-come-first-served ordering
  • Graceful failure: users who do not get the product see a clear "sold out" message

Tasks:

  • Calculate the QPS: 1M users / 60 seconds with 10 requests each = ? How does this compare to normal operations?
  • Design the inventory counter: single Redis counter vs pre-partitioned counters (1,000 units x 10 partitions). What are the trade-offs?
  • How do you prevent users from adding to cart after stock reaches zero? Where in the stack does the check happen?
  • Design the queueing mechanism: virtual waiting room that admits users gradually
# Pre-partitioned inventory approach:
# Split 10,000 units into 10 partitions of 1,000 each
# Each partition handled by a separate Redis instance
# User hash determines which partition they check
# When a partition exhausts, overflow to others

# This distributes the hot partition problem:
# Single counter: 166K writes/sec on ONE key
# 10 partitions: 16.6K writes/sec per key (manageable)

Exercise 2: Ride Matching Optimization

Improve the basic nearest-driver matching algorithm for Uber by incorporating multiple factors:

  • Distance to pickup (primary factor)
  • Driver rating (prefer higher-rated drivers)
  • Driver acceptance rate (prefer drivers who accept more rides)
  • Estimated time in traffic (not just straight-line distance)

Tasks:

  • Design a scoring function that weights these factors. How do you normalize different scales (km vs rating vs percentage)?
  • Implement the geohash-based candidate search: query the target cell plus 8 neighbors
  • How do you handle the case where no drivers are available in nearby cells? Expanding ring search?
  • Benchmark: how many geohash lookups per match request? At 700 QPS, is this sustainable?

Exercise 3: Booking Calendar Stress Test

Simulate 100 concurrent booking attempts for overlapping dates on the same property.

# Scenario:
# Property "Beach Villa" available Dec 24-31
# 100 guests simultaneously try to book Dec 25-27
# Only ONE should succeed. The rest should get
# "dates unavailable" without any race condition.

# Test with different locking strategies:
# 1. Pessimistic lock (lock dates first, then pay)
# 2. Optimistic lock (check-and-set with version number)
# 3. Queue-based (serialize all requests through a queue)

# Measure for each:
# - Correctness: exactly 1 booking succeeds?
# - Latency: how long does the winning user wait?
# - Fairness: does the first requester always win?

Tasks:

  • Implement pessimistic and optimistic locking strategies
  • Run the simulation 1,000 times and verify zero double-bookings
  • Compare average latency for the successful booking under each strategy
  • Which strategy degrades most gracefully under extreme contention (1,000 concurrent requests)?
