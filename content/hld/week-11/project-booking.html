Project: Booking System Prototype

This project implements a calendar-based booking system with per-listing per-date availability, double-booking prevention via pessimistic locking, a booking saga (lock -> pay -> confirm), dynamic pricing, search with date filtering, and cancellation policies with refunds.

from enum import Enum

Calendar Service with Pessimistic Locking

class CalendarService:
    """Manages availability with pessimistic locking."""
    LOCK_TTL_SEC = 600  # 10 minute lock timeout

    def __init__(self):
        self.calendar = {}  # (listing_id, date) -> DaySlot

    def check_availability(self, listing_id, check_in, check_out):
        """Check if all dates in range are available."""
        current = check_in
        while current < check_out:
            slot = self.calendar.get((listing_id, current))
            if slot and (not slot.available or slot.locked_by):
                return False
            current += timedelta(days=1)
        return True

    def lock_dates(self, listing_id, check_in, check_out, booking_id):
        """Acquire locks on date range. Atomic operation."""
        # First verify all dates are lockable
        if not self.check_availability(listing_id, check_in, check_out):
            return False
        # Lock all dates
        current = check_in
        while current < check_out:
            slot = self.calendar.setdefault(
                (listing_id, current), DaySlot())
            slot.locked_by = booking_id
            current += timedelta(days=1)
        return True

Booking Saga

The booking process follows a saga pattern with compensating transactions:

  • Step 1 - Lock: Acquire date locks (10-minute TTL). If any date is taken, fail immediately.
  • Step 2 - Pay: Process payment. If payment fails, release all locks (compensate step 1).
  • Step 3 - Confirm: Mark dates as booked, finalize booking record. If this fails, refund payment and release locks.

Dynamic Pricing

  • Weekend premium: Friday and Saturday nights 20-30% above base price
  • Seasonal pricing: Peak season (summer, holidays) multiplier up to 2x
  • Demand-based: High occupancy in the area increases suggested price
  • Last-minute discount: Unfilled dates within 48 hours may get reduced pricing

Search with Availability

  • First pass: filter by city, capacity, amenities, and price range
  • Second pass: for each candidate listing, check calendar availability for requested dates
  • Return only listings that are both matching filters AND available for the full date range

Key Takeaways

  • Pessimistic locking prevents double-booking with deterministic behavior under contention
  • Lock TTL prevents orphaned locks when users abandon the booking flow
  • The saga pattern with compensating transactions ensures consistency across calendar, payment, and booking services
  • Dynamic pricing requires balancing host revenue optimization with guest price sensitivity
