Project: Ride Matching Simulator

This project implements geohash-based driver matching with spatial indexing, ETA estimation based on distance and traffic, surge pricing based on supply/demand, and real-time driver pool management.

Geohash Engine

import math

BASE32 = "0123456789bcdefghjkmnpqrstuvwxyz"

def geohash_encode(lat: float, lng: float, precision: int = 6) -> str:
    """Encode lat/lng to geohash string."""
    lat_range = [-90.0, 90.0]
    lng_range = [-180.0, 180.0]
    bits, is_lng = [], True
    while len(bits) < precision * 5:
        if is_lng:
            mid = (lng_range[0] + lng_range[1]) / 2
            if lng >= mid:
                bits.append(1); lng_range[0] = mid
            else:
                bits.append(0); lng_range[1] = mid
        else:
            mid = (lat_range[0] + lat_range[1]) / 2
            if lat >= mid:
                bits.append(1); lat_range[0] = mid
            else:
                bits.append(0); lat_range[1] = mid
        is_lng = not is_lng
    result = []
    for i in range(0, len(bits), 5):
        chunk = bits[i:i+5]
        idx = sum(b << (4-j) for j, b in enumerate(chunk))
        result.append(BASE32[idx])
    return "".join(result)

def haversine(lat1, lng1, lat2, lng2) -> float:
    """Distance in km between two coordinates."""
    R = 6371
    dlat = math.radians(lat2 - lat1)
    dlng = math.radians(lng2 - lng1)
    a = (math.sin(dlat/2)**2 +
         math.cos(math.radians(lat1)) *
         math.cos(math.radians(lat2)) *
         math.sin(dlng/2)**2)
    return R * 2 * math.asin(math.sqrt(min(1, a)))

Matching Algorithm

The matching process works in stages:

  • Stage 1 - Candidate discovery: Compute geohash of pickup location, query drivers in the same cell plus 8 neighboring cells
  • Stage 2 - Distance filtering: Calculate haversine distance to each candidate, filter by max radius (e.g., 5 km)
  • Stage 3 - Scoring: Rank candidates by weighted score combining distance, rating, and acceptance history
  • Stage 4 - Offer: Send ride offer to top-scored driver. If declined or timed out, offer to next candidate.

Surge Pricing

  • City divided into zones (geohash-based)
  • Each zone tracks active drivers (supply) and recent ride requests (demand)
  • Surge multiplier = demand_count / supply_count, bounded between 1.0x and 3.0x
  • Recalculated every 2-5 minutes to reflect current conditions

Simulation Results

The simulator generates drivers distributed across a city grid and processes ride requests to measure:

  • Match success rate: Percentage of rides successfully matched
  • Average match distance: How far the matched driver is from pickup
  • Average ETA: Estimated time for driver to reach pickup
  • Surge distribution: Which zones experience surge and at what multiplier
  • Driver utilization: Percentage of time drivers are on active rides vs idle

Key Takeaways

  • Geohash enables O(1) lookup for nearby drivers instead of scanning all drivers
  • Precision 6 geohash (~1 km cells) is the sweet spot for urban ride matching
  • Surge pricing naturally balances supply and demand by incentivizing drivers to enter high-demand zones
  • The expanding ring search (try adjacent cells if no drivers found) handles sparse areas gracefully
