Service Discovery and Communication

How do microservices find each other and communicate? This lesson covers service-to-service communication patterns, service discovery mechanisms, and the API gateway pattern.

Communication Patterns

  • Synchronous (Request-Response): Service A calls B and waits. Protocols: REST (HTTP/JSON), gRPC (HTTP/2, protobuf). Use when the caller needs the result to proceed
  • Asynchronous (Event-Driven): Service A publishes a message, B processes it later. Protocols: Kafka, RabbitMQ, SQS. Use when the caller doesn't need an immediate result

In practice, most systems use both: sync for reads and user-facing responses, async for side effects and background processing.

Service Discovery

Service A needs to call Service B, but B has 5 instances on different IPs. Which IP does A use? And what happens when instances start or stop?

from typing import Dict, List, Optional
import random

class ServiceRegistry:
    """Simplified service registry (like Consul or Eureka)."""

    def __init__(self):
        self.services: Dict[str, List[str]] = {}  # name -> [addresses]
        self.health: Dict[str, bool] = {}

    def register(self, name: str, address: str):
        """Service registers itself on startup."""
        self.services.setdefault(name, []).append(address)
        self.health[address] = True

    def deregister(self, name: str, address: str):
        """Service deregisters on shutdown."""
        if name in self.services:
            self.services[name] = [a for a in self.services[name]
                                    if a != address]

    def discover(self, name: str) -> Optional[str]:
        """Client discovers a healthy instance (round-robin)."""
        instances = [a for a in self.services.get(name, [])
                     if self.health.get(a, False)]
        return random.choice(instances) if instances else None

Three Discovery Approaches

  • Client-Side Discovery (Eureka, Consul): Client queries registry, gets list of IPs, picks one. Client handles load balancing
  • Server-Side Discovery (AWS ALB, Kubernetes Services): Load balancer sits between client and services. Client calls one stable endpoint
  • DNS-Based Discovery: Service names resolve to IPs via DNS. Simple but DNS caching can cause stale routing

REST vs gRPC

  • REST: HTTP/JSON. Human-readable. Ubiquitous. Higher latency due to JSON serialization
  • gRPC: HTTP/2 + Protocol Buffers. Binary format. 2-10x faster. Streaming support. Requires code generation
  • Use REST for external/public APIs; use gRPC for internal service-to-service communication

API Gateway Pattern

A single entry point for all external clients. Routes requests to the appropriate microservice, handles cross-cutting concerns.

  • Authentication: Verify JWT tokens before requests reach services
  • Rate limiting: Throttle abusive clients
  • Request routing: Route /users/* to UserService, /orders/* to OrderService
  • Response aggregation: Combine responses from multiple services into one
  • Examples: Kong, AWS API Gateway, Nginx, Envoy

Backend for Frontend (BFF) Pattern

Each client type (web, mobile, IoT) gets its own API gateway. The web BFF aggregates data differently than the mobile BFF, optimizing payloads for each platform.
