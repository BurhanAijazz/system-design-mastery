Monolith vs Microservices

The most debated architectural decision in system design. The key insight: start with a monolith, split when you have a reason. Understanding the trade-offs is critical for both interviews and real-world systems.

Monolith Architecture

Everything in one application, one database, one deployment.

  • Simple deployment: one binary, one server
  • Easy debugging: one process, one stack trace
  • No network overhead: modules communicate via function calls
  • ACID transactions: one database covers all data
  • Simple development: one repo, one build

Downsides: scaling bottleneck (must scale everything together), team coupling, technology lock-in, deploy-everything, one bad module can crash the whole app.

Microservices Architecture

Each service owns its domain, its data, and its deployment.

# Monolith: all modules share one process and database
class Monolith:
    def __init__(self):
        self.db = SharedDatabase()
        self.user_module = UserModule(self.db)
        self.order_module = OrderModule(self.db)
        self.payment_module = PaymentModule(self.db)

# Microservices: each service has its own database
class UserService:
    def __init__(self):
        self.db = UserDatabase()  # Own database

class OrderService:
    def __init__(self):
        self.db = OrderDatabase()  # Own database

class PaymentService:
    def __init__(self):
        self.db = PaymentDatabase()  # Own database

Microservices Pros and Cons

  • Pro: Independent scaling (scale only what's bottlenecked)
  • Pro: Independent deployment (deploy one service without touching others)
  • Pro: Technology diversity (use the best language/DB per service)
  • Pro: Team autonomy (each team owns a service end-to-end)
  • Con: Network complexity (services talk over the network -- latency, failures)
  • Con: Data consistency (no cross-service transactions, need sagas)
  • Con: Operational overhead (need service discovery, monitoring, tracing)
  • Con: Debugging is harder (distributed tracing needed)

When to Split the Monolith

  • Team scaling: More than ~8-10 developers working in the same codebase
  • Different scaling needs: One module needs 100x more resources than others
  • Independent release cycles: One team deploys daily, another deploys monthly
  • Technology requirements: One module needs Python ML, another needs Go for performance

The Modular Monolith

A middle ground: keep one deployable unit but enforce strict module boundaries internally. Modules communicate through well-defined interfaces, not shared database tables. This gives you monolith simplicity with the option to extract services later.

Migration Strategy

  • Strangler Fig Pattern: Gradually route traffic from the monolith to new microservices. The monolith shrinks over time
  • Extract the service with the clearest boundary first (often notifications or analytics)
  • Never do a "big bang" rewrite
