Distributed Consensus: Raft and Paxos

How do distributed systems agree on a value when nodes can fail? Leader election, log replication, and the split-brain problem. You rarely implement Raft from scratch, but you must understand the concept.

The Consensus Problem

N nodes must agree on a single value, even when some nodes crash or network partitions occur.

  • Leader election: Who is the primary database node? Two leaders = split brain = data corruption
  • Distributed locks: Only one service should process a job. Two holders = duplicate processing
  • Configuration: All nodes must agree on cluster membership
  • Replicated state machine: All replicas apply operations in the same order

The Split-Brain Problem

A network partition splits the cluster into two groups. Both groups elect a leader. Two leaders accept writes that conflict.

Solution: majority quorum. You need more than N/2 votes to become leader. Only one partition can have a majority. This is why clusters use odd numbers: 3, 5, 7 nodes.

from enum import Enum
from typing import List, Dict, Optional

class Role(Enum):
    FOLLOWER = "follower"
    CANDIDATE = "candidate"
    LEADER = "leader"

class RaftNode:
    """Simplified Raft node for understanding consensus."""
    def __init__(self, node_id: str, peers: List[str]):
        self.id = node_id
        self.peers = peers
        self.role = Role.FOLLOWER
        self.current_term = 0
        self.voted_for: Optional[str] = None
        self.log: List[dict] = []

    def start_election(self):
        """Candidate requests votes from peers."""
        self.current_term += 1
        self.role = Role.CANDIDATE
        self.voted_for = self.id
        votes = 1  # Vote for self

        for peer in self.peers:
            # Request vote from peer
            # peer grants vote if:
            #   1. peer hasn't voted this term
            #   2. candidate's log is at least as up-to-date
            vote_granted = self._request_vote(peer)
            if vote_granted:
                votes += 1

        majority = (len(self.peers) + 1) // 2 + 1
        if votes >= majority:
            self.role = Role.LEADER

Raft Algorithm

Designed to be understandable (unlike Paxos). Used in etcd, CockroachDB, TiKV, Consul.

  • Three roles: Leader (handles requests), Follower (receives replication), Candidate (during election)
  • Leader Election: Followers become candidates after an election timeout. Request votes from peers. Majority wins
  • Log Replication: Leader appends entries to its log, replicates to followers. Entry is committed when a majority confirms
  • Safety: A node can only become leader if its log contains all committed entries

Paxos vs Raft

  • Paxos: The original consensus algorithm. Proven correct but notoriously difficult to understand and implement
  • Raft: Equivalent safety guarantees but decomposed into understandable sub-problems (leader election, log replication, safety)
  • In practice: use Raft (via etcd or Consul), not raw Paxos

Practical Consensus Systems

  • etcd: Raft-based KV store. Used by Kubernetes for cluster state
  • ZooKeeper: ZAB protocol (Paxos variant). Used by Kafka, Hadoop
  • Consul: Raft-based service discovery and configuration
