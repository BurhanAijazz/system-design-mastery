Saga Pattern: Distributed Transactions

Microservices can't use traditional database transactions across services. The Saga pattern solves this with local transactions + compensating actions. This is one of the most asked topics in system design interviews.

The Problem: No ACID Across Services

In a monolith, one transaction covers everything. In microservices, each service has its own database. You can't wrap a single transaction across 3 databases. Two-Phase Commit (2PC) works but is slow and brittle.

# Monolith: one transaction
# BEGIN TRANSACTION
#   INSERT INTO orders (...)
#   UPDATE inventory SET stock = stock - 1
#   INSERT INTO payments (...)
# COMMIT  -- all or nothing

# Microservices: each service has its own DB
# OrderService  -> orders_db
# InventoryService -> inventory_db
# PaymentService -> payment_db
# Can't wrap one transaction across 3 databases!

Saga Pattern Overview

A saga is a sequence of local transactions. Each step has a compensating transaction that undoes it on failure.

  • Step 1: Create Order | Compensate: Cancel Order
  • Step 2: Reserve Inventory | Compensate: Release Inventory
  • Step 3: Process Payment | Compensate: Refund Payment
  • Step 4: Confirm and Ship | (final step, no compensation needed)
from dataclasses import dataclass
from typing import List, Callable, Optional

@dataclass
class SagaStep:
    name: str
    action: Callable         # forward action
    compensation: Callable   # undo action
    success_rate: float = 0.9

class SagaOrchestrator:
    """Execute a saga with automatic compensation on failure."""

    def __init__(self, steps: List[SagaStep]):
        self.steps = steps

    def execute(self, context: dict) -> dict:
        completed = []
        for step in self.steps:
            success = step.action(context)
            if success:
                completed.append(step)
            else:
                # Compensate in reverse order
                for prev_step in reversed(completed):
                    prev_step.compensation(context)
                return {"status": "failed", "failed_at": step.name}

        return {"status": "completed"}

Choreography vs Orchestration

  • Choreography: Each service listens for events and decides what to do next. No central coordinator. Simpler for small flows but hard to track complex sagas
  • Orchestration: A central orchestrator service directs the saga steps. Easier to understand, monitor, and debug. Preferred for complex multi-step flows

Key Insights About Compensating Transactions

  • Compensating transactions are not simple undo. "Refund" is not "un-payment" -- it's a new transaction
  • "Release inventory" must handle the case where stock was already allocated to another order
  • Compensations can also fail -- need retry logic and manual intervention as a fallback
  • Some actions are not reversible (sent email, dispatched physical goods). Design for this

Saga Guarantees

  • Sagas guarantee eventual consistency, not immediate consistency
  • During execution, the system is in a temporarily inconsistent state (order exists but payment not yet processed)
  • This is acceptable for most business processes -- just make sure the user sees appropriate status updates
