Project: Notification System

A full LLD implementation of a multi-channel notification system. This project uses Strategy for delivery channels, Template Method for notification formatting, Observer for event-driven notifications, Builder for construction, Factory for channel creation, and Chain of Responsibility for validation.

Requirements

Functional:

  • Send notifications via multiple channels (Email, SMS, Push, InApp)
  • User preferences: which channels to receive notifications on
  • Notification templates with variable substitution
  • Priority levels (low, normal, high, critical)
  • Rate limiting per user per channel
  • Deduplication (do not send the same notification twice)
  • Retry on failure
  • Event-driven: listen for events and auto-send notifications

Out of scope: Actual email/SMS sending, rich media, scheduled notifications.

Core Entities

class Channel(Enum):
    EMAIL = "email"
    SMS = "sms"
    PUSH = "push"
    IN_APP = "in_app"

class Priority(Enum):
    LOW = 0
    NORMAL = 1
    HIGH = 2
    CRITICAL = 3

class NotificationStatus(Enum):
    PENDING = "pending"
    SENT = "sent"
    FAILED = "failed"
    BLOCKED = "blocked"

class ChannelPreference(Flag):
    NONE = 0
    EMAIL = auto()
    SMS = auto()
    PUSH = auto()
    IN_APP = auto()
    ALL = EMAIL | SMS | PUSH | IN_APP

Design Patterns Used

  • Strategy -- DeliveryChannel implementations (EmailChannel, SMSChannel, PushChannel, InAppChannel) each handle sending through their specific medium
  • Template Method -- NotificationTemplate defines the structure (subject, body) with variable placeholders like {user_name} that get substituted at send time
  • Observer -- EventBus listens for domain events (OrderPlaced, PaymentReceived) and triggers the appropriate notification templates
  • Builder -- NotificationBuilder provides a fluent API to construct notifications with recipient, channel, template, and priority
  • Factory -- ChannelFactory creates the appropriate DeliveryChannel from a Channel enum value
  • Chain of Responsibility -- Validation chain: PreferenceHandler (user wants this channel?) -> RateLimitHandler (too many recent sends?) -> DeduplicationHandler (already sent?) -> DeliveryHandler (actually send)

Implementation Steps

  • Define entities: Notification, Channel, Priority, Template, UserPreference
  • Implement delivery channel strategies for each channel type
  • Build the template system with variable substitution
  • Create the validation chain (preference -> rate limit -> dedup -> delivery)
  • Build the NotificationService that orchestrates sending
  • Add the EventBus for event-driven notification triggers
  • Implement retry logic with exponential backoff
