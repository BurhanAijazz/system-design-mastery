Pub/Sub and Message Queue Patterns

Publish-Subscribe and message queue architectures decouple producers from consumers. This lesson covers topic-based pub/sub, message queues, delivery guarantees, dead letter queues, fan-out, and acknowledgment with retry.

Message Model

A Message carries data between publisher and subscriber, with metadata for routing and reliability:

class MessageStatus(Enum):
    PENDING = "pending"
    DELIVERED = "delivered"
    ACKNOWLEDGED = "acknowledged"
    FAILED = "failed"
    DEAD = "dead"  # moved to dead letter queue


class Priority(Enum):
    LOW = 0
    NORMAL = 1
    HIGH = 2
    CRITICAL = 3


@dataclass
class Message:
    message_id: str
    topic: str
    payload: Any
    priority: Priority = Priority.NORMAL
    status: MessageStatus = MessageStatus.PENDING
    created_at: datetime = field(default_factory=datetime.now)
    retry_count: int = 0
    max_retries: int = 3

Subscriber Interface

Subscribers receive messages from topics they subscribe to. Two delivery styles exist:

  • Push-based -- the broker calls the subscriber's on_message() method (Observer pattern)
  • Pull-based -- the subscriber calls the broker's poll() method
class Subscriber(ABC):
    def __init__(self, subscriber_id: str):
        self.subscriber_id = subscriber_id

    @abstractmethod
    def on_message(self, message: Message) -> bool:
        """Process message. Return True if handled successfully."""
        pass


class PrintSubscriber(Subscriber):
    def on_message(self, message):
        print(f"[{self.subscriber_id}] Received: {message.payload}")
        return True

Topic and Broker

A Topic manages a set of subscribers and a message queue. The MessageBroker manages multiple topics and routes messages:

  • publish(topic, payload) -- create a message and deliver to all subscribers of that topic
  • subscribe(topic, subscriber) -- register a subscriber for a topic
  • unsubscribe(topic, subscriber) -- remove a subscriber

Delivery Guarantees

  • At-most-once -- fire and forget, fastest but messages can be lost
  • At-least-once -- retry until acknowledged, messages may be delivered multiple times
  • Exactly-once -- hardest to achieve, requires idempotency or deduplication

Dead Letter Queue

When a message fails delivery after max retries, it moves to the dead letter queue (DLQ) instead of being lost. This allows operators to inspect failed messages and replay them after fixing the issue.

Fan-Out Pattern

A single published message is delivered to all subscribers of a topic. This is the core of pub/sub -- one publisher, many consumers, fully decoupled.

Acknowledgment and Retry

For at-least-once delivery, the subscriber must acknowledge each message. If acknowledgment fails or times out, the broker retries delivery up to max_retries. After exhausting retries, the message goes to the DLQ.

Key Design Takeaways

  • Pub/Sub decouples publishers from subscribers -- neither knows about the other
  • Observer pattern at scale: pub/sub is the distributed version of Observer
  • Choose delivery guarantees based on the domain: notifications can be at-most-once, payments need at-least-once
  • Dead letter queues prevent message loss during failures
  • Priority queues ensure critical messages are processed first
