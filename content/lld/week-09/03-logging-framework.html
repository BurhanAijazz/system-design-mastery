Logging Framework Design

A logging framework is a real-world systems design problem that tests understanding of multiple patterns: Singleton for the logger registry, Strategy for formatters, Chain of Responsibility for filters, Observer for handlers, and Builder for configuration.

Log Levels

Log levels form a hierarchy using IntEnum so they support comparison. A logger set to WARNING will only process WARNING, ERROR, and CRITICAL -- lower levels are filtered out.

class LogLevel(IntEnum):
    DEBUG = 10
    INFO = 20
    WARNING = 30
    ERROR = 40
    CRITICAL = 50

Log Record

A LogRecord captures all information about a single log event:

@dataclass
class LogRecord:
    level: LogLevel
    message: str
    logger_name: str
    timestamp: datetime = field(default_factory=datetime.now)
    extra: Dict = field(default_factory=dict)

Formatter (Strategy Pattern)

Formatters convert a LogRecord into a string. Different handlers may use different formatters:

class Formatter(ABC):
    @abstractmethod
    def format(self, record: LogRecord) -> str:
        pass

class SimpleFormatter(Formatter):
    def format(self, record):
        return f"[{record.level.name}] {record.message}"

class JSONFormatter(Formatter):
    def format(self, record):
        return json.dumps({
            "level": record.level.name,
            "message": record.message,
            "logger": record.logger_name,
            "timestamp": record.timestamp.isoformat()
        })

class DetailedFormatter(Formatter):
    def format(self, record):
        return (f"{record.timestamp:%Y-%m-%d %H:%M:%S} "
                f"[{record.level.name}] "
                f"{record.logger_name}: {record.message}")

Handler Hierarchy (Observer Pattern)

Handlers receive log records and output them to a destination. Multiple handlers can be attached to a single logger:

  • ConsoleHandler -- prints to stdout
  • FileHandler -- writes to a file
  • RotatingFileHandler -- writes to a file with size-based rotation
  • MemoryHandler -- stores in memory (useful for testing)

Filter Chain (Chain of Responsibility)

Filters decide whether a log record should be processed. They form a chain -- each filter either passes the record through or blocks it:

  • LevelFilter -- blocks records below a threshold level
  • KeywordFilter -- blocks or allows records containing specific keywords
  • RateFilter -- limits how many records per second pass through (prevents log flooding)

Logger Registry (Singleton)

A global registry manages named loggers. Calling get_logger("app.db") always returns the same logger instance. Logger names support hierarchy: "app.db" inherits configuration from "app" unless overridden.

Thread Safety

Logging happens from multiple threads. Use a threading lock around handler output to prevent interleaved log lines.

Key Design Takeaways

  • IntEnum for log levels enables natural comparison (WARNING > INFO)
  • Strategy pattern for formatters -- same record, different output formats
  • Chain of Responsibility for filters -- compose filtering rules without modifying logger code
  • Singleton for the registry ensures consistent logger access across the application
  • Builder pattern simplifies logger configuration (set level, add handlers, set formatter)
