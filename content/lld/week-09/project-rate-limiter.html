Project: Rate Limiter System

A full LLD implementation of a configurable rate limiting system. This project uses Strategy for algorithm selection, Factory for creating limiters from configuration, Decorator for transparently applying rate limits, and Chain of Responsibility for composing multiple rules.

Requirements

Functional:

  • Multiple rate limiting algorithms (Token Bucket, Sliding Window, Fixed Window)
  • Configurable limits per rule (max requests, window size)
  • Per-client tracking (by client_id)
  • Composed limiters (multiple rules: 10/sec AND 1000/hour)
  • Rate limit any function via decorator
  • Rate limit result with remaining count and retry-after
  • Rule scopes: per-user, per-endpoint, global

Out of scope: Distributed rate limiting (Redis-based), IP-based tracking, response header injection.

Core Entities

class Algorithm(Enum):
    TOKEN_BUCKET = "token_bucket"
    SLIDING_WINDOW = "sliding_window"
    FIXED_WINDOW = "fixed_window"

class Scope(Enum):
    GLOBAL = "global"         # One limit for all clients
    PER_USER = "per_user"     # Separate limit per client_id
    PER_ENDPOINT = "per_endpoint"

@dataclass
class RateLimitRule:
    name: str
    max_requests: int
    window_seconds: float
    algorithm: Algorithm = Algorithm.TOKEN_BUCKET
    scope: Scope = Scope.PER_USER

@dataclass
class RateLimitResult:
    allowed: bool
    remaining: int
    retry_after: Optional[float] = None
    rule_name: Optional[str] = None

Design Patterns Used

  • Strategy -- Each algorithm (TokenBucketLimiter, SlidingWindowLimiter, FixedWindowLimiter) implements the RateLimiter interface. The RateLimitService selects the right strategy based on the rule's algorithm field
  • Factory -- RateLimiterFactory.create(rule) returns the appropriate limiter instance based on the Algorithm enum
  • Decorator -- The @rate_limited decorator wraps any function to automatically check rate limits before execution, raising RateLimitExceeded if blocked
  • Chain of Responsibility -- ComposedLimiter chains multiple rules together. A request must pass ALL rules to be allowed. The first failing rule determines the retry_after value

Key Implementation Detail

class RateLimitService:
    """Manages rules and checks rate limits."""

    def __init__(self):
        self._rules: List[RateLimitRule] = []
        self._limiters: Dict[str, RateLimiter] = {}

    def add_rule(self, rule: RateLimitRule) -> None:
        self._rules.append(rule)
        self._limiters[rule.name] = RateLimiterFactory.create(rule)

    def check(self, client_id: str) -> RateLimitResult:
        for rule in self._rules:
            limiter = self._limiters[rule.name]
            key = self._build_key(client_id, rule)
            if not limiter.allow_request(key):
                return RateLimitResult(
                    allowed=False, remaining=0,
                    rule_name=rule.name
                )
        return RateLimitResult(allowed=True, remaining=-1)

Implementation Steps

  • Define entities: Algorithm, Scope, RateLimitRule, RateLimitResult
  • Implement three algorithm strategies: Token Bucket, Sliding Window, Fixed Window
  • Build the RateLimiterFactory to create limiters from rules
  • Create the RateLimitService that manages rules and checks requests
  • Implement the ComposedLimiter for layered protection
  • Add the @rate_limited decorator for transparent application
  • Handle scope-based key generation (global, per-user, per-endpoint)
