Visitor Pattern

The Visitor pattern defines new operations on a structure of objects without modifying the classes themselves. The operation is moved into a separate "visitor" class. This uses double dispatch: the element calls the visitor, passing itself, so the correct visitor method is called based on both the element type and the visitor type.

The Problem: Adding Operations to Existing Classes

class CircleNaive:
    def __init__(self, radius: float):
        self.radius = radius

    def area(self) -> float:
        return math.pi * self.radius ** 2

    def draw(self) -> str:
        return f"Drawing circle (r={self.radius})"

    def to_json(self) -> str:
        return f'{{"type": "circle", "radius": {self.radius}}}'

    # Every new operation = modify this class.
    # What about perimeter? SVG export? Bounding box?
    # This violates OCP.

With 5 shapes and 10 operations, you get 50 methods scattered across shape classes. Every new operation requires modifying every shape class.

Solution: Visitor Pattern

Element Interface and Concrete Elements

from abc import ABC, abstractmethod
from dataclasses import dataclass
import math

class Shape(ABC):
    """Element -- accepts a visitor."""

    @abstractmethod
    def accept(self, visitor: "ShapeVisitor") -> None:
        """Double dispatch: call visitor.visit_<type>(self)."""
        pass


@dataclass
class Circle(Shape):
    radius: float

    def accept(self, visitor: "ShapeVisitor") -> None:
        visitor.visit_circle(self)


@dataclass
class Rectangle(Shape):
    width: float
    height: float

    def accept(self, visitor: "ShapeVisitor") -> None:
        visitor.visit_rectangle(self)


@dataclass
class Triangle(Shape):
    base: float
    height: float

    def accept(self, visitor: "ShapeVisitor") -> None:
        visitor.visit_triangle(self)

Visitor Interface

class ShapeVisitor(ABC):
    """Visitor interface with a visit method per element type."""

    @abstractmethod
    def visit_circle(self, circle: Circle) -> None: pass

    @abstractmethod
    def visit_rectangle(self, rect: Rectangle) -> None: pass

    @abstractmethod
    def visit_triangle(self, tri: Triangle) -> None: pass

Concrete Visitors (New Operations)

class AreaCalculator(ShapeVisitor):
    """Visitor that calculates area for each shape."""

    def __init__(self):
        self.total_area = 0.0

    def visit_circle(self, circle: Circle) -> None:
        area = math.pi * circle.radius ** 2
        self.total_area += area

    def visit_rectangle(self, rect: Rectangle) -> None:
        area = rect.width * rect.height
        self.total_area += area

    def visit_triangle(self, tri: Triangle) -> None:
        area = 0.5 * tri.base * tri.height
        self.total_area += area


class JSONExporter(ShapeVisitor):
    """Visitor that exports shapes to JSON."""

    def __init__(self):
        self.result = []

    def visit_circle(self, circle: Circle) -> None:
        self.result.append(f'{{"type": "circle", "radius": {circle.radius}}}')

    def visit_rectangle(self, rect: Rectangle) -> None:
        self.result.append(
            f'{{"type": "rectangle", "width": {rect.width}, "height": {rect.height}}}'
        )

    def visit_triangle(self, tri: Triangle) -> None:
        self.result.append(
            f'{{"type": "triangle", "base": {tri.base}, "height": {tri.height}}}'
        )

Usage: New Operations Without Changing Shapes

shapes = [Circle(5), Rectangle(4, 6), Triangle(3, 8)]

# Calculate total area
calc = AreaCalculator()
for shape in shapes:
    shape.accept(calc)
print(f"Total area: {calc.total_area:.2f}")

# Export to JSON -- no changes to shape classes!
exporter = JSONExporter()
for shape in shapes:
    shape.accept(exporter)
print(exporter.result)

Double Dispatch Explained

Python normally uses single dispatch: the method called depends on the object type. With the Visitor pattern, the method called depends on two types:

  1. The element type (Circle, Rectangle) determines which visit_* method is called
  2. The visitor type (AreaCalculator, JSONExporter) determines what that method does

When to Use

  • Stable class hierarchy with frequently changing operations (shapes, AST nodes)
  • Reporting and export -- generate different output formats
  • Code analysis -- lint rules, type checking, optimization passes
  • Tax/pricing calculations -- different calculation strategies over a product catalog

When NOT to Use

If the class hierarchy changes frequently (new shape types added often), the Visitor pattern becomes painful because every visitor must be updated. Use Visitor when elements are stable but operations change.
