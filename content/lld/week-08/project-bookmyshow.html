Project: BookMyShow

A full LLD implementation of a movie ticket booking platform. This project uses the State pattern for booking lifecycle, Strategy for seat pricing, Observer for booking notifications, and Factory for seat creation based on screen layout.

Requirements

Functional:

  • Browse movies by city
  • View theaters showing a movie
  • View available shows and seats
  • Select seats and book tickets
  • Different seat types with different prices (Regular, Premium, VIP)
  • Cancel booking with refund status
  • Search movies by name, genre, language
  • Notify user on booking confirmation

Out of scope: Payment gateway integration, food/beverage ordering, seat map UI rendering.

Core Entities

class Genre(Enum):
    ACTION = "action"
    COMEDY = "comedy"
    DRAMA = "drama"
    HORROR = "horror"
    SCIFI = "sci-fi"
    THRILLER = "thriller"

class SeatType(Enum):
    REGULAR = "regular"
    PREMIUM = "premium"
    VIP = "vip"

class SeatStatus(Enum):
    AVAILABLE = "available"
    HELD = "held"
    BOOKED = "booked"

class BookingStatus(Enum):
    PENDING = "pending"
    CONFIRMED = "confirmed"
    CANCELLED = "cancelled"

Architecture

The system is built around a BookingService that manages theaters, screens, shows, and bookings:

  • Theater -- has multiple Screens, belongs to a city
  • Screen -- has a layout of Seats, hosts Shows at specific times
  • Show -- links a Movie to a Screen at a specific time, tracks seat availability
  • Movie -- title, genre, language, duration, rating
  • Booking -- links a user to selected seats in a show, tracks status

Design Patterns Used

  • State -- Booking lifecycle: PENDING -> CONFIRMED -> CANCELLED. Each state defines valid transitions and operations
  • Strategy -- Seat pricing varies by type. RegularPricing, PremiumPricing, and VIPPricing each calculate prices differently
  • Observer -- When a booking is confirmed, observers (email service, SMS service) are notified
  • Factory -- Screen layout factory creates seats based on configuration (number of rows, seats per row, type mapping)

Key Implementation Considerations

# 1. User selects seats -> status changes to HELD
# 2. User confirms payment -> status changes to BOOKED
# 3. If payment times out -> status reverts to AVAILABLE
