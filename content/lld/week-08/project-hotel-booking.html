Project: Hotel Booking System

A full LLD implementation of a hotel room booking system. This project uses the State pattern for reservation lifecycle, Strategy for pricing, Observer for booking notifications, and Chain of Responsibility for reservation validation.

Requirements

Functional:

  • Search hotels by city, date range, room type
  • View available rooms for a hotel
  • Book a room (create reservation)
  • Check in and check out
  • Cancel reservation
  • Different room types (Standard, Deluxe, Suite)
  • Pricing strategies (seasonal, dynamic, promotional)
  • Guest reviews and ratings
  • Reservation validation chain

Out of scope: Payment gateway, loyalty program tiers, room service / amenities ordering.

Core Entities

class RoomType(Enum):
    STANDARD = "standard"
    DELUXE = "deluxe"
    SUITE = "suite"

class RoomStatus(Enum):
    AVAILABLE = "available"
    OCCUPIED = "occupied"
    MAINTENANCE = "maintenance"

class ReservationStatus(Enum):
    PENDING = "pending"
    CONFIRMED = "confirmed"
    CHECKED_IN = "checked_in"
    CHECKED_OUT = "checked_out"
    CANCELLED = "cancelled"

# Valid state transitions -- explicit guard against invalid moves
VALID_TRANSITIONS = {
    ReservationStatus.PENDING: {ReservationStatus.CONFIRMED,
                                 ReservationStatus.CANCELLED},
    ReservationStatus.CONFIRMED: {ReservationStatus.CHECKED_IN,
                                   ReservationStatus.CANCELLED},
    ReservationStatus.CHECKED_IN: {ReservationStatus.CHECKED_OUT},
    ReservationStatus.CHECKED_OUT: set(),
    ReservationStatus.CANCELLED: set(),
}

Architecture

The HotelService orchestrates hotels, rooms, reservations, and validation:

  • Hotel -- has multiple Rooms, belongs to a city, has reviews
  • Room -- has a type, status, base price, and floor number
  • Reservation -- links a guest to a room for a date range, tracks status through its lifecycle
  • ReservationValidator -- Chain of Responsibility: checks availability, then credit, then loyalty status

Design Patterns Used

  • State -- Reservation lifecycle: PENDING -> CONFIRMED -> CHECKED_IN -> CHECKED_OUT (or CANCELLED from PENDING/CONFIRMED). State transitions are validated against a whitelist
  • Strategy -- Pricing varies by season, demand, or promotion. SeasonalPricing adjusts rates by time of year, DynamicPricing adjusts by occupancy rate, PromotionalPricing applies discount codes
  • Observer -- Booking events trigger email and SMS notifications to guests
  • Chain of Responsibility -- Reservation requests pass through a validation chain: AvailabilityHandler -> CreditHandler -> LoyaltyHandler. Each handler either approves and forwards, or rejects

Key Implementation Considerations

# Date-range availability checking:
# A room is available for dates [check_in, check_out] if no
# existing reservation overlaps that range.

def is_available(self, room, check_in, check_out):
    for reservation in room.reservations:
        if (check_in < reservation.check_out and
                check_out > reservation.check_in):
            return False  # overlap detected
    return True
