Transaction and Ledger Patterns

Splitwise, payment systems, and booking platforms all need to track financial transactions. This lesson covers transaction modeling, the append-only ledger pattern, split algorithms (equal, exact, percentage), balance calculation from transaction history, and debt simplification.

Transaction Model

A transaction records a financial event: who paid, how much, who owes what, when, and the current status:

class TransactionStatus(Enum):
    PENDING = "pending"
    SETTLED = "settled"
    CANCELLED = "cancelled"


@dataclass
class Split:
    """How much a single user owes in an expense."""
    user_id: str
    amount: float


@dataclass
class Transaction:
    transaction_id: str
    description: str
    payer_id: str
    total_amount: float
    splits: List[Split]
    status: TransactionStatus = TransactionStatus.PENDING
    created_at: datetime = field(default_factory=datetime.now)
    group_id: Optional[str] = None

    def settle(self) -> None:
        self.status = TransactionStatus.SETTLED

    def cancel(self) -> None:
        self.status = TransactionStatus.CANCELLED

Split Strategies

Different ways to split an expense, each implemented as a Strategy:

  • Equal -- divide evenly among all participants, handle rounding remainder
  • Exact -- each person's share is specified explicitly
  • Percentage -- each person pays a percentage of the total
  • Share-based -- proportional to shares (e.g., 2:1:1)
class SplitStrategy(ABC):
    @abstractmethod
    def calculate(self, total: float, participants: List[str],
                  **kwargs) -> List[Split]:
        pass


class EqualSplit(SplitStrategy):
    def calculate(self, total, participants, **kwargs):
        share = round(total / len(participants), 2)
        splits = [Split(uid, share) for uid in participants]
        # Handle rounding: give remainder to first participant
        diff = round(total - share * len(participants), 2)
        if diff != 0:
            splits[0].amount += diff
        return splits


class PercentageSplit(SplitStrategy):
    def calculate(self, total, participants, **kwargs):
        percentages = kwargs.get("percentages", {})
        return [
            Split(uid, round(total * percentages[uid] / 100, 2))
            for uid in participants
        ]

Append-Only Ledger

The ledger pattern records every financial event as an immutable entry. You never update or delete entries -- only append new ones. To reverse a transaction, you add a compensating entry. This gives you a complete audit trail and makes balance calculation deterministic.

Balance Calculation

Balances are computed by scanning the ledger. For each transaction, the payer is owed money by each participant (excluding themselves). Net balances determine who owes whom:

  • Positive balance: others owe you money
  • Negative balance: you owe money to others

Debt Simplification

With N users, there can be up to N*(N-1)/2 debts. Simplification reduces the number of transactions needed to settle all debts by computing net balances and matching creditors with debtors. This is a greedy algorithm: sort creditors and debtors, then match the largest creditor with the largest debtor.

Key Design Takeaways

  • Use Strategy pattern for split calculation -- new split types require only a new strategy
  • Append-only ledgers provide complete audit trails and simplify debugging
  • Handle rounding carefully -- floating point errors accumulate in financial code
  • Debt simplification is a graph problem: compute net flows to minimize settlements
  • Track transaction status (PENDING, SETTLED, CANCELLED) as explicit state
