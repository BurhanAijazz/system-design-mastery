ATM Machine Walkthrough

This lesson walks through designing an ATM Machine using the 6-step LLD interview framework. The ATM is a classic State pattern problem with Chain of Responsibility for transaction processing and Strategy for cash dispensing.

Step 1: Clarify Requirements

  • "What operations should the ATM support?" -- Balance inquiry, withdrawal, deposit, and fund transfer
  • "Should I model the bank backend or just the ATM?" -- Focus on the ATM; assume a BankingService interface
  • "Should I handle cash denomination?" -- Yes, minimize the number of notes dispensed
  • "What about error cases?" -- Handle wrong PIN (3 attempts max), insufficient funds, ATM out of cash

Functional Requirements:

  • Insert card, authenticate with PIN (3 attempts max)
  • Balance inquiry, cash withdrawal, cash deposit, fund transfer
  • Print receipt, eject card
  • Handle errors: wrong PIN, insufficient funds, ATM out of cash

Out of scope: Cheque deposit, card-less transactions, multi-language support, network failure handling.

Step 2: Identify Entities

# ATM State Machine:
# Idle -> CardInserted -> PINEntered -> TransactionSelected
#      -> Processing -> Dispensing -> Idle

class ATMStatus(Enum):
    IDLE = "idle"
    CARD_INSERTED = "card_inserted"
    PIN_ENTERED = "pin_entered"
    TRANSACTION_SELECTED = "transaction_selected"
    PROCESSING = "processing"
    DISPENSING = "dispensing"
    OUT_OF_SERVICE = "out_of_service"

class TransactionType(Enum):
    BALANCE_INQUIRY = "balance_inquiry"
    WITHDRAWAL = "withdrawal"
    DEPOSIT = "deposit"
    TRANSFER = "transfer"

class CashDenomination(Enum):
    HUNDRED = 100
    FIFTY = 50
    TWENTY = 20
    TEN = 10
    FIVE = 5
    ONE = 1

Step 3: Key Components

  • ATM -- id, location, state, cash_dispenser, card_reader, screen, keypad
  • Card -- card_number, pin_hash, account_id
  • Account -- id, holder_name, balance, account_type
  • BankingService -- interface for verify_pin, get_balance, withdraw, deposit, transfer
  • CashDispenser -- tracks denominations and available stock, dispenses cash
  • Transaction -- id, type, account, amount, status, timestamp

Step 4: Design Patterns

  • State -- Each ATM state (Idle, CardInserted, PINEntered, etc.) is a separate class with its own valid operations. In Idle state, only insertCard() works. In PINEntered, only selectTransaction() works. Invalid operations in any state raise clear errors
  • Chain of Responsibility -- Transaction validation: AuthHandler (is PIN correct?) -> BalanceCheckHandler (sufficient funds?) -> WithdrawalLimitHandler (within daily limit?) -> Each handler approves and forwards, or rejects
  • Strategy -- Cash dispensing: greedy algorithm that minimizes the number of notes by using the largest denominations first, checking available stock of each denomination
  • Command -- Each transaction (withdraw, deposit, transfer) is a command object that can be logged and potentially reversed

Step 5: Cash Dispensing Algorithm

# Greedy approach: use largest denominations first
def dispense(self, amount: int) -> Dict[CashDenomination, int]:
    result = {}
    remaining = amount
    for denom in sorted(CashDenomination,
                        key=lambda d: d.value, reverse=True):
        if remaining <= 0:
            break
        count = min(remaining // denom.value,
                    self._stock[denom])
        if count > 0:
            result[denom] = count
            remaining -= count * denom.value
            self._stock[denom] -= count
    if remaining > 0:
        raise ATMOutOfCashError("Cannot dispense exact amount")
    return result

Key Takeaway

The ATM is the definitive State pattern problem. When presenting in an interview, draw the state machine diagram first, then show how each state class restricts operations to only those valid in that state.
