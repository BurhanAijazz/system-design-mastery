Time Management and Communication

How to structure your 45-minute LLD interview, present your design verbally, and handle curveball questions. Mastering this framework is as important as knowing design patterns.

The 5-5-5-25-5 Framework

Structure your 45-minute interview into five phases:

  • Phase 1 (5 min) -- Requirements: Ask 5-8 clarifying questions, state assumptions explicitly, define scope (in/out), list functional requirements
  • Phase 2 (5 min) -- Entities and Relationships: Identify 6-10 core entities, key attributes for each, relationships (1:1, 1:N, N:M), quick ASCII UML diagram
  • Phase 3 (5 min) -- Interfaces and Patterns: Identify 3-5 key interfaces, select 2-4 patterns with justification, define API method signatures
  • Phase 4 (25 min) -- Implementation: Core entity classes (5 min), pattern implementations (10 min), main service/controller (10 min). Skip getters/setters, full error handling, utility methods
  • Phase 5 (5 min) -- Discussion: Walk through a use case, discuss trade-offs, mention what you would add with more time, brief scalability notes

Critical rule: If you are running out of time at 35 minutes, STOP implementing and move to discussion. A design that is 80% implemented but well-discussed beats 100% code with no context.

Verbal Presentation Templates

Use these phrases to communicate clearly at each phase:

# OPENING (Phase 1):
"Let me start by clarifying the requirements. I want to make
 sure I'm solving the right problem."

"Based on our discussion, here are my assumptions:
 [list 3-4]. Does that sound right?"

# ENTITIES (Phase 2):
"The core entities I see are [list them]. Let me draw
 a quick relationship diagram..."

# PATTERNS (Phase 3):
"For the [X] problem, I'm going to use the [Pattern]
 pattern because [reason]. An alternative would be [Y],
 but [trade-off]."

# IMPLEMENTATION (Phase 4):
"Let me start with the core entities, then build up
 to the service layer. I'll focus on the happy path first."

"I'm going to skip [X] for now and come back to it
 if we have time."

# DISCUSSION (Phase 5):
"Let me walk through a typical use case: [describe flow]"
"If I had more time, I would add [feature]."

When to Ask vs When to Assume

Always ask about:

  • Scope (what is in, what is out)
  • Scale (single machine vs distributed)
  • Main actors/users
  • Core operations expected
  • Specific business rules

Safe to assume (state your assumption):

  • Standard data types (string IDs, float amounts)
  • In-memory storage (unless told otherwise)
  • Single-threaded (unless concurrency is the focus)

Handling Mid-Implementation Questions

When the interviewer asks "What about X?" during coding, use one of three patterns:

  • Quick acknowledge: "Great point. I'd handle that by [brief approach]. Let me note it and come back after the core flow."
  • Pivot if important: "That's critical. Let me add an interface for it now." Spend 2 minutes, then continue.
  • Defer gracefully: "That's out of the scope we discussed, but I'd approach it with [brief idea]. Should I focus on that or continue?"

Pre-Submission Checklist

  • Are all core entities defined with key attributes?
  • Am I using enums for status, type, and category values (not raw strings)?
  • Are there clear interfaces/abstract classes for the key abstractions?
  • Did I justify each pattern choice with a reason and trade-off?
  • Can I walk through at least one complete use case end-to-end?
