6-Step LLD Interview Framework

A repeatable approach for any Low-Level Design interview question. Follow these steps in order, adjusting time based on the interview length.

Time Budget (45-minute interview)

  • Step 1: Clarify Requirements -- 2-3 min
  • Step 2: Identify Core Entities -- 2-3 min
  • Step 3: Define Interfaces/APIs -- 3-5 min
  • Step 4: Design Class Hierarchy -- 5-7 min
  • Step 5: Implement Core Logic -- 15-20 min
  • Step 6: Discuss Trade-offs -- 2-3 min

Step 1: Clarify Requirements

Before writing any code, ask clarifying questions. This shows maturity and prevents wasted effort.

Template questions:

  1. "What are the main use cases / features?"
  2. "How many users / scale should I design for?"
  3. "Should I handle concurrency / thread safety?"
  4. "What is in scope and what is NOT in scope?"
  5. "Are there specific design patterns you'd like to see?"

Split requirements into three categories:

  • Functional -- what the system does ("Users can create short URLs")
  • Non-functional -- how it performs ("Handle 1000 req/sec")
  • Out of scope -- explicitly exclude ("No user authentication")

Step 2: Identify Core Entities

Use the noun extraction technique: read each requirement and underline the nouns.

"VEHICLES can enter and exit the PARKING LOT"
  -> Vehicle, ParkingLot

"Different VEHICLE TYPES: CAR, MOTORCYCLE, TRUCK"
  -> Vehicle (abstract), Car, Motorcycle, Truck

"Different PARKING SPOT SIZES: SMALL, MEDIUM, LARGE"
  -> ParkingSpot, SpotSize (enum)

"Calculate PARKING FEE based on DURATION"
  -> ParkingTicket, FeeCalculator

Classify each noun:

  • If it has its own behavior -- Entity (class)
  • If it is just a value -- Attribute (field on a class)
  • If it is a fixed set of values -- Enum

Step 3: Define Interfaces / APIs

Define the public API before implementation. This forces you to think about how the system is used.

class ParkingLot:
    def park(self, vehicle: Vehicle) -> Optional[ParkingTicket]: ...
    def unpark(self, ticket: ParkingTicket) -> float: ...
    def available_spots(self) -> Dict[SpotType, int]: ...

class ParkingSpot(ABC):
    @abstractmethod
    def can_fit(self, vehicle: Vehicle) -> bool: ...
    def park(self, vehicle: Vehicle) -> None: ...
    def release(self) -> None: ...

Step 4: Design Class Hierarchy

Draw the UML diagram showing classes, relationships, and patterns:

  • Use inheritance for "is-a" (Vehicle -> Car, Truck)
  • Use composition for "owns" (ParkingLot owns ParkingSpots)
  • Use enums for fixed categories (VehicleType, SpotSize)
  • Apply design patterns where appropriate (Strategy for fee calculation)

Step 5: Implement Core Logic

Start with the most important classes. Implement the "happy path" first, then edge cases.

class ParkingLot:
    def __init__(self, floors: List[ParkingFloor], rate: PricingStrategy):
        self._floors = floors
        self._rate = rate
        self._tickets: Dict[str, ParkingTicket] = {}

    def park(self, vehicle: Vehicle) -> Optional[ParkingTicket]:
        for floor in self._floors:
            spot = floor.find_spot(vehicle)
            if spot:
                spot.park(vehicle)
                ticket = ParkingTicket(vehicle, spot, datetime.now())
                self._tickets[ticket.ticket_id] = ticket
                return ticket
        return None  # lot is full

Step 6: Discuss Trade-offs

After implementation, discuss your design choices:

  • Extensibility: "Adding a new vehicle type is easy -- just create a new subclass"
  • Patterns: "I used Strategy for pricing so we can swap algorithms without changing the lot"
  • Concurrency: "In a real system, I'd add locks around spot allocation"
  • Testing: "The abstract interfaces make it easy to mock in unit tests"

Common Mistakes

  • Jumping straight into code without clarifying requirements
  • Over-engineering with too many patterns
  • Ignoring edge cases (lot full, invalid ticket)
  • Not discussing trade-offs at the end
