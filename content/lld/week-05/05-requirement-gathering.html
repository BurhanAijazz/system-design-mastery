Requirement Gathering and API Design

Before writing a single line of implementation, you need two things: clear requirements (what to build) and clean API design (how users interact with it). This lesson provides templates for common LLD problems and shows how to design public APIs before implementation.

Requirement Gathering Template

For any LLD interview question, fill in this template:

SYSTEM: [name of system]
ACTORS: [who uses it?]

FUNCTIONAL REQUIREMENTS:
  1. [Actor] can [action] ...
  2. [Actor] can [action] ...

NON-FUNCTIONAL (if asked):
  - Scale: [number of users / requests]
  - Latency: [response time targets]
  - Durability: [data persistence needs]

OUT OF SCOPE:
  - [things you are NOT building]

ASSUMPTIONS:
  - [things you are assuming]

Example: Parking Lot Requirements

SYSTEM: Parking Lot Management System
ACTORS: Driver, Parking Attendant, Admin

FUNCTIONAL REQUIREMENTS:
  1. Driver can park a vehicle in an available spot
  2. Driver can retrieve their vehicle using a ticket
  3. System assigns the nearest suitable spot
  4. System calculates fee based on duration
  5. Different vehicle types: motorcycle, car, truck
  6. Different spot sizes: small, medium, large
  7. Admin can view occupancy status

OUT OF SCOPE:
  - Online reservation
  - Multiple parking floors (single level)
  - Payment gateway integration

ASSUMPTIONS:
  - One vehicle per spot
  - Single entry/exit point
  - Fee calculated per hour (rounded up)

Example: Movie Ticket Booking Requirements

SYSTEM: Movie Ticket Booking System
ACTORS: Customer, Theater Admin, System

FUNCTIONAL REQUIREMENTS:
  1. Customer can search movies by name, city, or genre
  2. Customer can view available shows for a movie
  3. Customer can select seats from a seating chart
  4. Customer can book tickets (hold -> confirm)
  5. Customer can cancel a booking
  6. Theater Admin can add movies, shows, and theaters
  7. System prevents double-booking of seats
  8. System holds seats temporarily during booking

OUT OF SCOPE:
  - Payment processing
  - User reviews and ratings
  - Notifications / emails

API Design Principles

Design the public interface before writing implementation:

  • Verb-based method names -- park(), book(), search()
  • Clear parameter types -- use enums, dataclasses, not raw strings
  • Return meaningful types -- Optional[Ticket] not bool
  • Fail explicitly -- raise exceptions or return None for error cases

API Design: Parking Lot

from abc import ABC, abstractmethod
from typing import List, Optional, Dict
from enum import Enum
from datetime import datetime

class VehicleType(Enum):
    CAR = "car"
    MOTORCYCLE = "motorcycle"
    TRUCK = "truck"

class SpotType(Enum):
    SMALL = "small"
    MEDIUM = "medium"
    LARGE = "large"


class ParkingLotAPI(ABC):
    """Public API for the parking lot system."""

    @abstractmethod
    def park_vehicle(self, vehicle_type: VehicleType,
                     license_plate: str) -> Optional["ParkingTicket"]:
        """Park a vehicle. Returns ticket or None if lot is full."""
        pass

    @abstractmethod
    def unpark_vehicle(self, ticket_id: str) -> float:
        """Unpark using ticket. Returns fee amount."""
        pass

    @abstractmethod
    def get_availability(self) -> Dict[SpotType, int]:
        """Return count of available spots per type."""
        pass

    @abstractmethod
    def get_occupancy_rate(self) -> float:
        """Return percentage of spots occupied."""
        pass

API Design: Movie Booking

class BookingAPI(ABC):
    @abstractmethod
    def search_movies(self, query: str, city: str) -> List["Movie"]:
        pass

    @abstractmethod
    def get_shows(self, movie_id: str, date: str) -> List["Show"]:
        pass

    @abstractmethod
    def get_available_seats(self, show_id: str) -> List["Seat"]:
        pass

    @abstractmethod
    def hold_seats(self, show_id: str,
                   seat_ids: List[str]) -> Optional["BookingHold"]:
        """Hold seats for 10 minutes while user completes payment."""
        pass

    @abstractmethod
    def confirm_booking(self, hold_id: str) -> "Booking":
        pass

    @abstractmethod
    def cancel_booking(self, booking_id: str) -> bool:
        pass

The Two-Phase Booking Pattern

Many booking systems use a two-phase approach to prevent double-booking:

  1. Hold: temporarily reserve the resource for a time window
  2. Confirm: finalize the reservation (usually after payment)

If the hold expires without confirmation, the resource is released automatically. This prevents both lost sales and double-bookings.

Tips for Interview API Design

  • Start with 3-5 core methods, not 20
  • Use abstract base classes to define contracts
  • Prefer return types over side effects
  • Think about what the caller needs, not what the system stores internally
