UML Class Diagram Basics

UML (Unified Modeling Language) class diagrams show the structure of a system: classes, their attributes, methods, and relationships. In LLD interviews, you often sketch class diagrams on a whiteboard. You do not need perfect UML, but you need to communicate structure clearly.

UML Class Notation

A UML class is drawn as a box with three sections:

+---------------------------+
|       ClassName           |   <- Class name
+---------------------------+
| + public_attr: str        |   <- Attributes
| - _private_attr: int      |
| # _protected_attr: float  |
+---------------------------+
| + public_method(): str    |   <- Methods
| - _private_method(): None |
| + static_method(): int    |
+---------------------------+

Visibility Symbols

  • + public -- no underscore in Python
  • - private -- single underscore _ in Python
  • # protected -- single underscore _ in Python (convention only)

Python does not have true private/protected access modifiers. It is all convention. In UML, use + for public API and - for internal implementation.

Example: BankAccount

UML:
+---------------------------+
|       BankAccount         |
+---------------------------+
| - _account_id: str        |
| - _owner: str             |
| - _balance: float         |
+---------------------------+
| + deposit(amount): None   |
| + withdraw(amount): bool  |
| + get_balance(): float    |
+---------------------------+

Translating directly to Python:

class BankAccount:
    """Maps directly from the UML diagram above."""

    def __init__(self, account_id: str, owner: str, balance: float = 0.0):
        self._account_id = account_id   # - private
        self._owner = owner             # - private
        self._balance = balance         # - private

    def deposit(self, amount: float) -> None:       # + public
        self._balance += amount

    def withdraw(self, amount: float) -> bool:      # + public
        if amount > self._balance:
            return False
        self._balance -= amount
        return True

    def get_balance(self) -> float:                 # + public
        return self._balance

Abstract Classes and Interfaces

UML uses stereotypes:
+----------------------------+
|  <<abstract>>              |
|    Shape                   |
+----------------------------+
| # _color: str              |
+----------------------------+
| + area(): float            |  (italic = abstract)
| + perimeter(): float       |  (italic = abstract)
| + describe(): str          |  (concrete)
+----------------------------+

from abc import ABC, abstractmethod

class Shape(ABC):
    def __init__(self, color: str = "black"):
        self._color = color

    @abstractmethod
    def area(self) -> float:
        pass

    @abstractmethod
    def perimeter(self) -> float:
        pass

    def describe(self) -> str:
        return f"{self._color} shape, area={self.area():.2f}"

Enums in UML

+-------------------+
| <<enum>>          |
|   SpotSize        |
+-------------------+
| SMALL             |
| MEDIUM            |
| LARGE             |
+-------------------+

from enum import Enum

class SpotSize(Enum):
    SMALL = "small"
    MEDIUM = "medium"
    LARGE = "large"

Static and Class Methods

In UML, static methods are underlined. In Python:

class MathUtils:
    @staticmethod
    def factorial(n: int) -> int:
        if n <= 1:
            return 1
        return n * MathUtils.factorial(n - 1)

    @classmethod
    def from_string(cls, value: str) -> "MathUtils":
        return cls()

Tips for LLD Interviews

  • Draw class boxes quickly -- name, key attributes, key methods
  • Skip trivial getters/setters in the diagram
  • Focus on relationships between classes, not exhaustive details
  • Use visibility markers (+, -) to show your design intent
  • Mark abstract classes and interfaces clearly
