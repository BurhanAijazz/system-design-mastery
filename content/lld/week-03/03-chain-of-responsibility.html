Chain of Responsibility Pattern

The Chain of Responsibility pattern passes a request along a chain of handlers. Each handler decides either to process the request or to pass it to the next handler in the chain.

The Problem: Monolithic Handler

Without this pattern, you end up with a single giant function handling all cases:

def process_support_ticket_naive(ticket: dict) -> str:
    """Giant if/elif that handles all levels -- hard to extend."""
    severity = ticket.get("severity", "low")

    if severity == "low":
        return f"Level 1 resolved: {ticket['issue']}"
    elif severity == "medium":
        return f"Level 2 resolved: {ticket['issue']}"
    elif severity == "high":
        return f"Manager resolved: {ticket['issue']}"
    elif severity == "critical":
        return f"Director resolved: {ticket['issue']}"
    else:
        return "Unhandled ticket!"

Adding a new severity level or handler requires modifying this function, violating the Open/Closed Principle.

Solution: Chain of Responsibility

Build a chain of handlers. Each handler checks if it can handle the request. If yes, it processes it. If no, it passes it along to the next handler.

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class SupportTicket:
    issue: str
    severity: str  # low, medium, high, critical
    customer: str


class SupportHandler(ABC):
    """Abstract handler with a link to the next handler."""

    def __init__(self):
        self._next_handler: Optional[SupportHandler] = None

    def set_next(self, handler: SupportHandler) -> SupportHandler:
        """Set the next handler and return it (for chaining)."""
        self._next_handler = handler
        return handler

    def handle(self, ticket: SupportTicket) -> str:
        """Try to handle. If can't, pass to next."""
        if self.can_handle(ticket):
            return self.process(ticket)
        elif self._next_handler:
            return self._next_handler.handle(ticket)
        else:
            return f"No handler available for: {ticket.issue}"

    @abstractmethod
    def can_handle(self, ticket: SupportTicket) -> bool:
        pass

    @abstractmethod
    def process(self, ticket: SupportTicket) -> str:
        pass

Concrete Handlers

class Level1Support(SupportHandler):
    """Handles low-severity tickets."""

    def can_handle(self, ticket: SupportTicket) -> bool:
        return ticket.severity == "low"

    def process(self, ticket: SupportTicket) -> str:
        return f"[L1 Support] Resolved '{ticket.issue}' for {ticket.customer}"


class Level2Support(SupportHandler):
    """Handles medium-severity tickets."""

    def can_handle(self, ticket: SupportTicket) -> bool:
        return ticket.severity == "medium"

    def process(self, ticket: SupportTicket) -> str:
        return f"[L2 Support] Resolved '{ticket.issue}' for {ticket.customer}"


class ManagerSupport(SupportHandler):
    """Handles high-severity tickets."""

    def can_handle(self, ticket: SupportTicket) -> bool:
        return ticket.severity == "high"

    def process(self, ticket: SupportTicket) -> str:
        return f"[Manager] Escalated and resolved '{ticket.issue}'"

Building and Using the Chain

# Build the chain: L1 -> L2 -> Manager
l1 = Level1Support()
l2 = Level2Support()
manager = ManagerSupport()

l1.set_next(l2).set_next(manager)

# All tickets enter at the start of the chain
ticket = SupportTicket("Slow WiFi", "low", "Alice")
print(l1.handle(ticket))    # L1 handles it

ticket = SupportTicket("Server down", "high", "Bob")
print(l1.handle(ticket))    # Passed through L1 -> L2 -> Manager

When to Use Chain of Responsibility

  • Middleware pipelines -- authentication, logging, CORS, rate limiting
  • Support ticket escalation -- L1 to L2 to manager to director
  • Request validation -- validate format, then auth, then permissions
  • Event handling -- GUI event bubbling from child to parent
  • Logging systems -- console handler, file handler, alert handler

Key Benefits

  • Adding a new handler is easy -- create a class, insert it into the chain
  • Each handler focuses on one responsibility
  • The sender does not need to know which handler will process the request
  • The chain order can be configured at runtime

Fluent Chain Building

The set_next() method returns the handler, enabling fluent chain construction:

chain = Level1Support()
chain.set_next(Level2Support()).set_next(ManagerSupport()).set_next(DirectorSupport())
