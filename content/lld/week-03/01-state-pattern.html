State Pattern

The State pattern allows an object to alter its behavior when its internal state changes. The object will appear to change its class. Instead of giant if/elif blocks checking state in every method, you encapsulate each state as a separate class.

The Problem: If/Elif State Hell

Consider a vending machine with multiple states. Without the State pattern, every method must check the current state with long conditional chains:

class VendingMachineNaive:
    """Every method checks the current state -- messy and error-prone."""

    def __init__(self):
        self.state = "idle"  # idle, has_money, dispensing, out_of_stock
        self.balance = 0.0
        self.stock = 3

    def insert_money(self, amount: float) -> str:
        if self.state == "idle":
            self.balance += amount
            self.state = "has_money"
            return f"Inserted ${amount:.2f}. Balance: ${self.balance:.2f}"
        elif self.state == "has_money":
            self.balance += amount
            return f"Added ${amount:.2f}. Balance: ${self.balance:.2f}"
        elif self.state == "dispensing":
            return "Please wait, dispensing in progress."
        elif self.state == "out_of_stock":
            return "Machine is out of stock. Returning money."
        return "Unknown state"

    def select_product(self, price: float) -> str:
        if self.state == "idle":
            return "Please insert money first."
        elif self.state == "has_money":
            if self.balance >= price:
                self.state = "dispensing"
                # ... more conditional logic
        # ... repeated for every state

Every method has the same if/elif chain. Adding a new state means modifying every method. This violates the Open/Closed Principle.

Solution: The State Pattern

Encapsulate each state as a separate class. Each state class handles behavior for that specific state and knows which state to transition to next.

Key Components

  • Context -- the object whose behavior changes (e.g., VendingMachine)
  • State Interface -- abstract class defining all possible actions
  • Concrete States -- one class per state, implementing behavior for that state
from abc import ABC, abstractmethod

class VendingState(ABC):
    """Abstract base state. Each concrete state handles behavior
    for that particular state of the vending machine."""

    @abstractmethod
    def insert_money(self, machine: "VendingMachine", amount: float) -> str:
        pass

    @abstractmethod
    def select_product(self, machine: "VendingMachine", price: float) -> str:
        pass

    @abstractmethod
    def eject_money(self, machine: "VendingMachine") -> str:
        pass

Concrete State Classes

class IdleState(VendingState):
    def insert_money(self, machine, amount):
        machine.balance += amount
        machine.set_state(HasMoneyState())
        return f"Inserted ${amount:.2f}"

    def select_product(self, machine, price):
        return "Please insert money first."

    def eject_money(self, machine):
        return "No money to return."


class HasMoneyState(VendingState):
    def insert_money(self, machine, amount):
        machine.balance += amount
        return f"Added ${amount:.2f}. Balance: ${machine.balance:.2f}"

    def select_product(self, machine, price):
        if machine.balance >= price:
            machine.balance -= price
            machine.stock -= 1
            machine.set_state(
                IdleState() if machine.stock > 0 else OutOfStockState()
            )
            return f"Dispensing! Change: ${machine.balance:.2f}"
        return f"Need ${price - machine.balance:.2f} more."

    def eject_money(self, machine):
        amount = machine.balance
        machine.balance = 0
        machine.set_state(IdleState())
        return f"Returning ${amount:.2f}"

The Context Class

class VendingMachine:
    """Context -- delegates behavior to the current state."""

    def __init__(self, stock: int = 5):
        self.balance = 0.0
        self.stock = stock
        self._state: VendingState = IdleState()

    def set_state(self, state: VendingState) -> None:
        self._state = state

    def insert_money(self, amount: float) -> str:
        return self._state.insert_money(self, amount)

    def select_product(self, price: float) -> str:
        return self._state.select_product(self, price)

    def eject_money(self) -> str:
        return self._state.eject_money(self)

When to Use the State Pattern

  • Vending machines -- idle, has money, dispensing, out of stock
  • Document workflows -- draft, review, approved, published
  • Order processing -- pending, paid, shipped, delivered
  • TCP connections -- listening, established, closed
  • Game characters -- idle, walking, running, jumping, attacking

Key Benefits

  • Adding a new state means adding a new class -- no existing code changes
  • Each state class is focused and easy to understand
  • State transitions are explicit and controlled
  • Eliminates complex conditional logic scattered across methods

State vs Strategy

Both patterns use polymorphism to delegate behavior, but they differ in intent:

  • State: The object changes its behavior automatically based on internal state transitions. The state knows about other states.
  • Strategy: The client explicitly selects an algorithm. Strategies are independent of each other.
