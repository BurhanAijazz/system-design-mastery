Open/Closed Principle (OCP)

The Open/Closed Principle states that software entities (classes, modules, functions) should be open for extension but closed for modification. In practical terms, you should be able to add new behavior without changing existing, tested code. This principle reduces the risk of introducing bugs when requirements evolve.

Why It Matters

Every time you modify existing code, you risk breaking something that already works. OCP encourages designs where new features are added by writing new code rather than editing old code. The most common technique to achieve this is the Strategy Pattern, where behavior is encapsulated behind an interface and injected at runtime.

Bad Example: if/elif Chains

Consider a discount calculator that uses conditional branching to determine discounts. Every time a new customer type is added, the existing function must be modified: 

# BAD: Violates OCP - must modify this function for every new customer type

class DiscountCalculator:
    def calculate_discount(self, customer_type: str, amount: float) -> float:
        if customer_type == "regular":
            return amount * 0.05
        elif customer_type == "premium":
            return amount * 0.10
        elif customer_type == "vip":
            return amount * 0.20
        elif customer_type == "employee":
            return amount * 0.30
        elif customer_type == "military":
            return amount * 0.15
        else:
            return 0.0

# Adding a new customer type requires modifying calculate_discount().
# This means retesting the entire function and risking regressions.

Problems with this approach:

  • Adding a new customer type forces changes to a working method.
  • The method grows unbounded as customer types increase.
  • Unit tests must be updated and re-run for the entire method each time.
  • Multiple developers adding types simultaneously cause merge conflicts.

Good Example: Strategy Pattern

We define an abstract DiscountStrategy interface and implement each customer type as a separate class. The calculator delegates to whichever strategy is provided, and new types require zero changes to existing code: 

from abc import ABC, abstractmethod


class DiscountStrategy(ABC):
    """Abstract interface for discount calculation strategies."""

    @abstractmethod
    def calculate(self, amount: float) -> float:
        pass


class RegularDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.05


class PremiumDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.10


class VIPDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.20


class EmployeeDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.30


class DiscountCalculator:
    """Calculator is CLOSED for modification, OPEN for extension."""

    def __init__(self, strategy: DiscountStrategy):
        self._strategy = strategy

    def calculate_discount(self, amount: float) -> float:
        return self._strategy.calculate(amount)


# Usage
calc = DiscountCalculator(VIPDiscount())
print(calc.calculate_discount(200.0))  # 40.0

Adding New Behavior Without Changing Existing Code

Suppose a new "military" discount is needed. We simply create a new class: 

# NEW FILE: no existing code is touched

class MilitaryDiscount(DiscountStrategy):
    def calculate(self, amount: float) -> float:
        return amount * 0.15


# Wire it up
calc = DiscountCalculator(MilitaryDiscount())
print(calc.calculate_discount(100.0))  # 15.0

Notice that DiscountCalculator was never modified. The new behavior was added purely through extension.

OCP with a Registry Pattern

For dynamic dispatch (e.g., mapping strings from config files to strategies), combine OCP with a registry: 

class DiscountRegistry:
    _strategies: dict[str, type[DiscountStrategy]] = {}

    @classmethod
    def register(cls, name: str, strategy_cls: type[DiscountStrategy]):
        cls._strategies[name] = strategy_cls

    @classmethod
    def get(cls, name: str) -> DiscountStrategy:
        if name not in cls._strategies:
            raise ValueError(f"Unknown discount type: {name}")
        return cls._strategies[name]()


# Register strategies (could be done via decorators or config)
DiscountRegistry.register("regular", RegularDiscount)
DiscountRegistry.register("premium", PremiumDiscount)
DiscountRegistry.register("vip", VIPDiscount)

# Usage from config
strategy = DiscountRegistry.get("premium")
calc = DiscountCalculator(strategy)
print(calc.calculate_discount(300.0))  # 30.0

Key Takeaways

  • OCP reduces regression risk by keeping working code untouched.
  • The Strategy Pattern is the most common way to achieve OCP.
  • Use abstract base classes or protocols to define extension points.
  • A registry can bridge the gap between string-based config and OCP-compliant strategies.
  • OCP does not mean you never modify code — it means you design so that common changes are extensions, not modifications.
