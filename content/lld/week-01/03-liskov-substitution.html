Liskov Substitution Principle (LSP)

The Liskov Substitution Principle states that objects of a superclass should be replaceable with objects of a subclass without altering the correctness of the program. If class B is a subclass of class A, then anywhere you use an instance of A, you should be able to substitute an instance of B and everything still works correctly.

The Formal Definition

Barbara Liskov originally stated: "If for each object o1 of type S there is an object o2 of type T such that for all programs P defined in terms of T, the behavior of P is unchanged when o1 is substituted for o2, then S is a subtype of T."

In simpler terms: subclasses must honor the contracts established by their parent classes.

The Classic Rectangle/Square Problem

The most famous LSP violation is the Rectangle/Square hierarchy. Mathematically, a square is a rectangle, so it seems natural to make Square inherit from Rectangle. But this breaks LSP.

Bad Example: Square Extends Rectangle

class Rectangle:
    def __init__(self, width: float, height: float):
        self._width = width
        self._height = height

    @property
    def width(self) -> float:
        return self._width

    @width.setter
    def width(self, value: float):
        self._width = value

    @property
    def height(self) -> float:
        return self._height

    @height.setter
    def height(self, value: float):
        self._height = value

    def area(self) -> float:
        return self._width * self._height


class Square(Rectangle):
    """A square IS-A rectangle... or is it?"""

    def __init__(self, side: float):
        super().__init__(side, side)

    @Rectangle.width.setter
    def width(self, value: float):
        # Must keep width == height for a square
        self._width = value
        self._height = value

    @Rectangle.height.setter
    def height(self, value: float):
        # Must keep width == height for a square
        self._width = value
        self._height = value

Why This Breaks LSP

Consider a function that works with any Rectangle: 

def resize_and_check(rect: Rectangle):
    """Set width to 5 and height to 10, then verify area."""
    rect.width = 5
    rect.height = 10
    expected_area = 50  # 5 * 10
    actual_area = rect.area()

    assert actual_area == expected_area, (
        f"Expected area {expected_area}, got {actual_area}"
    )

# Works fine with Rectangle
resize_and_check(Rectangle(1, 1))  # passes

# Fails with Square!
resize_and_check(Square(1))  # AssertionError: Expected area 50, got 100

When height is set to 10 on the Square, it also sets width to 10, resulting in area 100 instead of 50. The Square cannot be substituted for a Rectangle without breaking the program. This is a textbook LSP violation.

Good Example: Separate Shape Hierarchy

Instead of forcing an inheritance relationship, model both as independent implementations of a common interface: 

from abc import ABC, abstractmethod


class Shape(ABC):
    @abstractmethod
    def area(self) -> float:
        pass

    @abstractmethod
    def perimeter(self) -> float:
        pass


class Rectangle(Shape):
    def __init__(self, width: float, height: float):
        self._width = width
        self._height = height

    @property
    def width(self) -> float:
        return self._width

    @width.setter
    def width(self, value: float):
        self._width = value

    @property
    def height(self) -> float:
        return self._height

    @height.setter
    def height(self, value: float):
        self._height = value

    def area(self) -> float:
        return self._width * self._height

    def perimeter(self) -> float:
        return 2 * (self._width + self._height)


class Square(Shape):
    def __init__(self, side: float):
        self._side = side

    @property
    def side(self) -> float:
        return self._side

    @side.setter
    def side(self, value: float):
        self._side = value

    def area(self) -> float:
        return self._side ** 2

    def perimeter(self) -> float:
        return 4 * self._side

Now both classes honor the Shape contract independently. Any function that accepts a Shape works correctly with either: 

def print_shape_info(shape: Shape):
    print(f"Area: {shape.area()}")
    print(f"Perimeter: {shape.perimeter()}")

print_shape_info(Rectangle(5, 10))  # Area: 50, Perimeter: 30
print_shape_info(Square(7))          # Area: 49, Perimeter: 28

LSP Rules of Thumb

  • Preconditions: A subclass must not strengthen preconditions (require more).
  • Postconditions: A subclass must not weaken postconditions (promise less).
  • Invariants: A subclass must preserve all invariants of the parent.
  • History constraint: A subclass must not introduce state changes the parent would not allow.

Detecting LSP Violations

  • A subclass method raises NotImplementedError for an inherited method.
  • A subclass silently ignores or overrides setter behavior.
  • Client code uses isinstance() checks before calling methods.
  • Unit tests pass for the parent but fail when substituted with a child.

Key Takeaways

  • Inheritance should model behavioral compatibility, not just mathematical "IS-A" relationships.
  • If a subclass cannot fully honor the parent's contract, use composition or a shared interface instead.
  • LSP violations often surface as subtle, hard-to-debug failures in production code.
  • Design by contract (preconditions, postconditions, invariants) helps you verify LSP compliance.
