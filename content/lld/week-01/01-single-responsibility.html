Single Responsibility Principle (SRP)

"A class should have only one reason to change." — Robert C. Martin

The Problem

When a class handles multiple unrelated responsibilities, changing one feature can break another. This makes the code fragile and hard to test.

Bad Example: God Class

class UserManager:
    """Violates SRP - handles CRUD, email, AND reporting"""

    def create_user(self, name, email):
        # Database logic
        user = {"name": name, "email": email}
        self.save_to_db(user)
        self.send_welcome_email(user)
        self.generate_signup_report(user)
        return user

    def save_to_db(self, user):
        print(f"Saving {user['name']} to database...")

    def send_welcome_email(self, user):
        print(f"Sending welcome email to {user['email']}...")

    def generate_signup_report(self, user):
        print(f"Generating PDF report for {user['name']}...")

This class has three reasons to change: database schema changes, email template changes, or report format changes.

Good Example: Separated Responsibilities

from abc import ABC, abstractmethod

class UserRepository:
    """Handles ONLY persistence"""
    def save(self, user):
        print(f"Saving {user['name']} to database...")

class EmailService:
    """Handles ONLY email"""
    def send_welcome(self, user):
        print(f"Sending welcome email to {user['email']}...")

class ReportGenerator:
    """Handles ONLY reports"""
    def generate_signup_report(self, user):
        print(f"Generating PDF report for {user['name']}...")

class UserService:
    """Orchestrates user creation - single responsibility: coordination"""
    def __init__(self, repo, email, reports):
        self._repo = repo
        self._email = email
        self._reports = reports

    def create_user(self, name, email):
        user = {"name": name, "email": email}
        self._repo.save(user)
        self._email.send_welcome(user)
        self._reports.generate_signup_report(user)
        return user

How to Identify SRP Violations

  • The class name contains "And" or "Manager" (red flag)
  • You can describe the class using "and" — "It saves users and sends emails and generates reports"
  • Changes in unrelated features force changes to the same class
  • The class imports from many unrelated modules

Key Takeaways

  • Each class should have exactly one reason to change
  • SRP doesn't mean one method per class — it means one responsibility
  • Orchestrating other services is itself a valid single responsibility
  • SRP is the foundation all other SOLID principles build upon
